# Модуль 10. Итоговый проект

---

## Урок 10.1. Планирование проекта (например, To-Do List или погодное приложение)
**Время: 40 минут**

### Цель урока
Научиться формулировать техническое задание для итогового JavaScript-проекта, опираясь на темы, пройденные в курсе.

### 1. Выбор темы итогового проекта

Примеры тем, подходящих для итогового проекта:

- **To-Do List с сохранением в localStorage** — расширенная версия примера из Урока 6.6.
- **Погодное приложение** — получение данных о погоде через публичный API по названию города.
- **Приложение для поиска фильмов/книг** — обращение к публичному API с результатами и карточками.
- **Калькулятор бюджета** — учёт доходов/расходов с сохранением в localStorage и подсчётом статистики.
- **Квиз (тест с вопросами)** — с подсчётом баллов и таймером на каждый вопрос.

### 2. Структура технического задания

1. **Какую задачу решает приложение?**
2. **Какие функции должны быть реализованы?**
3. **Какие темы курса будут использованы?** (DOM, события, fetch, localStorage и т.д.)
4. **Как будут храниться данные?** (в переменных на время сессии, в localStorage, через API)

### 3. Пример технического задания

**Проект:** Погодное приложение

**Задача:** Позволить пользователю узнать текущую погоду в любом городе мира.

**Функции:**
- Поле ввода названия города и кнопка поиска.
- Загрузка данных о погоде через публичный API.
- Отображение температуры, описания погоды и иконки.
- История последних 5 поисковых запросов (сохранённая в localStorage).
- Обработка ошибок (город не найден, нет соединения).

**Используемые темы курса:** DOM, события, fetch/async-await, JSON, localStorage, обработка ошибок.

### 4. Оценка объёма задачи

Итоговый проект должен быть реализуем за 3–5 часов (время модуля 10) и явно опираться минимум на 4–5 тем из курса. Не стоит замахиваться на слишком масштабную идею — лучше сделать компактное приложение хорошо, чем масштабное — наполовину.

### 5. Работа с localStorage (пригодится для большинства проектов)

```javascript
// Сохранение данных
localStorage.setItem("history", JSON.stringify(["Алматы", "Астана"]));

// Загрузка данных
let history = JSON.parse(localStorage.getItem("history")) || [];

// Удаление данных
localStorage.removeItem("history");
```

### 6. Домашнее задание к уроку

Составь техническое задание для своего итогового проекта по образцу из пункта 3: задача, функции, используемые темы курса, способ хранения данных.

### Резюме урока

- Итоговый проект должен опираться на реальные темы курса: DOM, события, асинхронность, работу с API.
- Хорошее техническое задание чётко определяет функции и способ хранения данных до начала кодирования.
- `localStorage` — удобный способ сохранять данные пользователя между сессиями без использования сервера.

## Урок 10.2. Проектирование структуры (HTML/CSS/JS раздельно)
**Время: 55 минут**

### Цель урока
Спроектировать структуру файлов и функций будущего приложения до начала написания основной логики.

### 1. Разделение на файлы

Для итогового проекта рекомендуется чёткое разделение ответственности между файлами:

```
weather_app/
│
├── index.html    # разметка страницы
├── style.css      # оформление
└── script.js       # логика приложения
```

Такое разделение (структура/оформление/поведение) — стандартная практика веб-разработки.

### 2. Проектирование HTML-разметки

```html
<!DOCTYPE html>
<html lang="ru">
<head>
    <meta charset="UTF-8">
    <title>Погода</title>
    <link rel="stylesheet" href="style.css">
</head>
<body>
    <div id="app">
        <h1>Узнай погоду</h1>
        <input type="text" id="city-input" placeholder="Введите город">
        <button id="search-btn">Поиск</button>
        <div id="weather-result"></div>
        <ul id="history-list"></ul>
    </div>
    <script src="script.js"></script>
</body>
</html>
```

### 3. Проектирование функций через заглушки (stubs)

Прежде чем реализовывать полную логику, полезно набросать структуру функций:

```javascript
// script.js

async function getWeather(city) {
    // получить данные о погоде через API
}

function displayWeather(data) {
    // отобразить данные о погоде на странице
}

function displayError(message) {
    // отобразить сообщение об ошибке
}

function saveToHistory(city) {
    // сохранить город в историю поиска (localStorage)
}

function renderHistory() {
    // отобразить историю поиска на странице
}

function init() {
    // назначить обработчики событий, запустить приложение
}

init();
```

### 4. Схема взаимодействия функций

```
init()
  │
  └──> обработчик клика на #search-btn
         │
         ├──> getWeather(city) ──> displayWeather(data) / displayError(message)
         │
         └──> saveToHistory(city) ──> renderHistory()
```

Такая схема помогает заранее увидеть, в каком порядке вызываются функции и какие данные передаются между ними.

### 5. Проектирование данных

```javascript
// Пример структуры данных, которые вернёт API погоды (условно)
{
    city: "Алматы",
    temperature: 24,
    description: "Ясно",
    icon: "☀️"
}
```

Полезно заранее продумать, в каком виде приложение будет хранить и передавать данные между функциями — это упрощает написание кода на следующем этапе.

### 6. CSS — минимальное оформление

На этапе проектирования достаточно определить общую структуру стилей, не углубляясь в детали дизайна — основной фокус итогового модуля на JavaScript-логике, а не на визуальном оформлении.

### Резюме урока

- Разделение на HTML/CSS/JS файлы — стандартная практика структурирования веб-проекта.
- Проектирование функций через заглушки до написания реальной логики помогает продумать архитектуру приложения заранее.
- Полезно заранее описать структуру данных, с которыми будет работать приложение.

## Урок 10.3. Реализация основного функционала
**Время: 90 минут**

### Цель урока
Реализовать рабочую версию итогового проекта на основе технического задания и спроектированной структуры.

### 1. Рекомендуемый порядок реализации

1. Реализовать функцию получения данных (`fetch`/API).
2. Реализовать функцию отображения результата.
3. Связать функции с событиями пользовательского интерфейса.
4. Добавить дополнительный функционал (история, localStorage).
5. Проверять работу после реализации каждой функции.

### 2. Пример реализации получения данных

```javascript
const API_KEY = "your_api_key";

async function getWeather(city) {
    const response = await fetch(
        `https://api.openweathermap.org/data/2.5/weather?q=${city}&appid=${API_KEY}&units=metric&lang=ru`
    );
    
    if (!response.ok) {
        throw new Error("Город не найден");
    }
    
    const data = await response.json();
    
    return {
        city: data.name,
        temperature: Math.round(data.main.temp),
        description: data.weather[0].description
    };
}
```

> Примечание: для реального проекта потребуется зарегистрироваться на выбранном сервисе погоды и получить собственный API-ключ — у большинства подобных сервисов есть бесплатный тариф, достаточный для учебных целей.

### 3. Реализация отображения результата

```javascript
function displayWeather(weather) {
    const container = document.querySelector("#weather-result");
    container.innerHTML = `
        <h2>${weather.city}</h2>
        <p>${weather.temperature}°C</p>
        <p>${weather.description}</p>
    `;
}

function displayError(message) {
    const container = document.querySelector("#weather-result");
    container.innerHTML = `<p class="error">${message}</p>`;
}
```

### 4. Реализация истории поиска

```javascript
function saveToHistory(city) {
    let history = JSON.parse(localStorage.getItem("history")) || [];
    
    if (!history.includes(city)) {
        history.unshift(city);
        history = history.slice(0, 5);   // храним только последние 5 запросов
        localStorage.setItem("history", JSON.stringify(history));
    }
    
    renderHistory();
}

function renderHistory() {
    let history = JSON.parse(localStorage.getItem("history")) || [];
    const list = document.querySelector("#history-list");
    
    list.innerHTML = history
        .map(city => `<li class="history-item">${city}</li>`)
        .join("");
}
```

### 5. Связывание всех функций через обработчики событий

```javascript
async function handleSearch() {
    const input = document.querySelector("#city-input");
    const city = input.value.trim();
    
    if (city === "") return;
    
    try {
        const weather = await getWeather(city);
        displayWeather(weather);
        saveToHistory(weather.city);
    } catch (error) {
        displayError(error.message);
    }
}

document.querySelector("#search-btn").addEventListener("click", handleSearch);

document.querySelector("#history-list").addEventListener("click", function(event) {
    if (event.target.classList.contains("history-item")) {
        document.querySelector("#city-input").value = event.target.textContent;
        handleSearch();
    }
});

renderHistory();   // отображаем сохранённую историю при загрузке страницы
```

### 6. Постепенная проверка

После реализации каждой функции стоит сразу протестировать её отдельно (например, вызвать `getWeather("Алматы")` напрямую в консоли), прежде чем переходить к следующей — это значительно упрощает поиск ошибок.

### Резюме урока

- Реализацию стоит начинать с функции получения данных, а не с интерфейса.
- Разделение на отдельные функции (получение данных, отображение, сохранение истории) делает код организованным и облегчает отладку.
- Постоянная проверка каждой части сразу после реализации экономит время на поиске ошибок в конце разработки.

## Урок 10.4. Тестирование и отладка
**Время: 60 минут**

### Цель урока
Проверить работу итогового проекта на разных сценариях использования и исправить найденные ошибки.

### 1. Ручное тестирование по сценариям

- Поиск существующего города — отображаются ли корректные данные?
- Поиск несуществующего/некорректного города — показывается ли сообщение об ошибке?
- Повторный поиск одного и того же города — не дублируется ли он в истории?
- Клик по городу из истории — повторно ли запускается поиск?
- Перезагрузка страницы — сохраняется ли история поиска?

### 2. Проверка граничных случаев

- Что произойдёт при пустом вводе (нажатие кнопки без ввода города)?
- Что произойдёт при отсутствии интернет-соединения?
- Что произойдёт, если API временно недоступен (сервер вернул 500)?
- Корректно ли обрабатываются спецсимволы или пробелы в названии города?

### 3. Использование DevTools для отладки

- Вкладка **Console** — проверка ошибок и отладочных `console.log()`.
- Вкладка **Network** — просмотр реальных сетевых запросов к API, их статуса и содержимого ответа.
- Вкладка **Application → Local Storage** — просмотр сохранённых данных истории поиска.

### 4. Типичные ошибки, которые стоит проверить

```javascript
// Забыли await перед асинхронной операцией
async function getWeather(city) {
    let response = fetch(url);   // ошибка: забыт await, response - это Promise, а не данные
    let data = response.json();  // ошибка
}

// Не обработана ошибка сети
async function getWeather(city) {
    let response = await fetch(url);
    let data = await response.json();   // упадёт без try/catch при сетевой ошибке
}
```

### 5. Улучшение пользовательского опыта

По результатам тестирования полезно добавить:
- Индикатор загрузки на время запроса к API.
- Отключение кнопки поиска на время выполнения запроса (чтобы избежать повторных кликов).
- Понятные, дружелюбные сообщения об ошибках вместо технических деталей.

### 6. Чек-лист готовности проекта

- [ ] Все функции из технического задания реализованы.
- [ ] Обработаны основные ошибки (пустой ввод, город не найден, сетевая ошибка).
- [ ] Данные корректно сохраняются в localStorage между перезагрузками страницы.
- [ ] Код разделён на логические функции, а не находится в одном большом блоке.
- [ ] В консоли браузера нет ошибок при обычном использовании приложения.

### Резюме урока

- Тестирование должно охватывать как типичные сценарии, так и граничные, нестандартные случаи.
- Вкладки Console и Network в DevTools — основные инструменты отладки клиентских веб-приложений.
- Улучшение пользовательского опыта (индикаторы загрузки, понятные ошибки) — важная часть доведения проекта до готового состояния.

## Урок 10.5. Презентация проекта
**Время: 40 минут**

### Цель урока
Подготовить итоговый проект к демонстрации: финально оформить код и подготовить структуру презентации.

### 1. Финальное оформление кода

- Убедиться, что все переменные и функции названы понятно и в стиле camelCase.
- Убрать отладочные `console.log()`, оставленные во время разработки.
- Проверить, что код не содержит закомментированных «мёртвых» участков от экспериментов.
- Добавить краткий `README.md` с описанием проекта и инструкцией по запуску.

### 2. Пример README.md

```markdown
# Погодное приложение

Веб-приложение для просмотра текущей погоды в любом городе мира.

## Возможности
- Поиск текущей погоды по названию города
- История последних 5 поисковых запросов (сохраняется в localStorage)
- Обработка ошибок при некорректном вводе или отсутствии соединения

## Как запустить
Открыть файл index.html в браузере (рекомендуется через расширение Live Server).
```

### 3. Структура презентации проекта

1. **Идея.** Какую задачу решает приложение? (1 минута)
2. **Демонстрация.** Показать приложение в работе, включая обработку ошибок. (2–3 минуты)
3. **Технические детали.** Какие темы курса использованы (DOM, fetch, localStorage и т.д.), какие были сложности. (2 минуты)
4. **Возможные улучшения.** Что можно добавить при наличии больше времени (например, переключение единиц измерения температуры, прогноз на неделю). (1 минута)

### 4. Частые вопросы, к которым стоит подготовиться

- Как обрабатываются ошибки сети или некорректный ввод пользователя?
- Почему были выбраны именно эти структуры данных для хранения истории?
- Как приложение справляется с одновременными или повторными запросами пользователя?
- Как можно было бы адаптировать приложение под React или другой фреймворк?

### 5. Значение итогового проекта

Итоговый проект демонстрирует не отдельные изолированные темы, а умение объединять их в цельное работающее приложение: DOM для отображения, события для интерактивности, асинхронность и API для получения реальных данных, localStorage для сохранения состояния между сессиями.

### 6. Что дальше после курса

- **Фреймворки.** Изучение React, Vue или Angular для более масштабной фронтенд-разработки.
- **TypeScript.** Надстройка над JavaScript, добавляющая статическую типизацию — популярна в реальных проектах.
- **Node.js.** Использование JavaScript для серверной разработки.
- **Углублённая работа с API.** Аутентификация, работа с REST и GraphQL API.

### Резюме урока

- Перед презентацией важно привести код в аккуратный вид и подготовить краткое описание проекта.
- Презентация строится по схеме: идея → демонстрация → технические детали → возможные улучшения.
- Итоговый проект показывает умение объединять разрозненные темы курса в единое работающее приложение — это ключевой навык веб-разработчика.
