# Модуль 1. Основы Git

## Урок 1.1. Зачем нужен контроль версий, установка и настройка Git
**Время: 35 минут**

### Цель урока
Понять назначение систем контроля версий, установить и настроить Git для дальнейшей работы.

### 1. Проблема, которую решает контроль версий

Без системы контроля версий разработчики часто хранят историю изменений вручную — через папки вида `project_final`, `project_final_v2`, `project_final_ИСПРАВЛЕНО`, `project_final_ТОЧНО_ПОСЛЕДНЯЯ`. Такой подход неудобен, ненадёжен и делает практически невозможной совместную работу нескольких человек над одним и тем же кодом.

**Git** — самая распространённая в мире распределённая система контроля версий, которая отслеживает все изменения в файлах проекта, позволяет вернуться к любой предыдущей версии и даёт возможность нескольким разработчикам параллельно работать над одним проектом без потери изменений друг друга.

### 2. Почему Git — стандарт индустрии

- Позволяет видеть полную историю изменений: кто, когда и что именно изменил, и почему (через сообщения коммитов).
- Поддерживает параллельную разработку через ветки без риска потерять чужие изменения.
- Работает преимущественно локально — большинство операций (просмотр истории, создание веток, коммиты) не требуют подключения к интернету.
- Является основой практически всех современных практик командной разработки и автоматизации (CI/CD).

### 3. Git vs GitHub — важное различие

Важно не путать два понятия: **Git** — это сама технология контроля версий, работающая локально на компьютере. **GitHub** (а также аналоги — GitLab, Bitbucket) — это облачный сервис для хранения Git-репозиториев и совместной работы над ними через интернет. Git можно использовать и без GitHub, но GitHub не может существовать без Git — он построен поверх этой технологии.

### 4. Установка Git

**Windows:** скачать установщик с официального сайта git-scm.com и установить с настройками по умолчанию (устанавливается вместе с удобным терминалом Git Bash).

**macOS:**
```bash
brew install git
```

**Linux (Ubuntu/Debian):**
```bash
sudo apt update
sudo apt install git
```

Проверка установки:

```bash
git --version
```

### 5. Первоначальная настройка Git

После установки необходимо указать своё имя и email — эта информация будет прикрепляться к каждому сделанному изменению (коммиту):

```bash
git config --global user.name "Алина Иванова"
git config --global user.email "alina@example.com"
```

Флаг `--global` означает, что эти настройки будут применяться ко всем Git-репозиториям на этом компьютере, а не только к одному конкретному проекту.

### 6. Проверка текущих настроек

```bash
git config --list
```

### 7. Основная терминология Git

| Термин | Значение |
|---|---|
| Репозиторий (repository) | папка проекта, отслеживаемая Git |
| Коммит (commit) | сохранённая «точка» в истории изменений с описанием |
| Ветка (branch) | независимая линия разработки внутри репозитория |
| Remote (удалённый репозиторий) | копия репозитория, хранящаяся на сервере (например, GitHub) |

### Резюме урока

- Git — распределённая система контроля версий, отслеживающая изменения в файлах проекта и позволяющая нескольким разработчикам работать параллельно.
- Важно различать Git (саму технологию) и GitHub (облачный сервис для хранения репозиториев, построенный на основе Git).
- После установки Git необходимо настроить имя и email через `git config --global` — эта информация прикрепляется к каждому коммиту.

## Урок 1.2. Создание репозитория: git init, git add, git commit
**Время: 40 минут**

### Цель урока
Освоить базовый рабочий цикл Git: создание репозитория и сохранение изменений через коммиты.

### 1. Создание нового репозитория: git init

```bash
mkdir my_project
cd my_project
git init
```

Команда `git init` создаёт в текущей папке скрытую служебную папку `.git`, в которой Git хранит всю историю изменений проекта. С этого момента Git начинает отслеживать изменения файлов в этой папке.

### 2. Три состояния файлов в Git

- **Untracked (неотслеживаемый)** — файл существует в папке проекта, но Git ещё не отслеживает его изменения.
- **Staged (проиндексированный)** — файл помечен для включения в следующий коммит.
- **Committed (зафиксированный)** — изменения файла сохранены в постоянной истории репозитория.

### 3. Проверка состояния: git status

```bash
git status
```

Эта команда — одна из самых часто используемых в Git. Она показывает, какие файлы изменены, какие уже добавлены в индекс (staged), а какие ещё вообще не отслеживаются.

### 4. Добавление файлов в индекс: git add

```bash
git add file.txt          # добавить в индекс конкретный файл
git add file1.txt file2.txt   # добавить сразу несколько конкретных файлов
git add .                   # добавить ВСЕ изменённые и новые файлы в текущей папке и подпапках
```

### 5. Сохранение изменений: git commit

```bash
git commit -m "Добавлен файл конфигурации проекта"
```

Флаг `-m` позволяет указать сообщение коммита прямо в самой команде. Хорошее сообщение коммита кратко и понятно объясняет, что именно было изменено и, желательно, зачем — это становится особенно ценным при просмотре истории проекта спустя месяцы или годы.

### 6. Практический пример: полный рабочий цикл

```bash
mkdir demo_project
cd demo_project
git init

echo "# Мой первый проект" > README.md
git status                                    # README.md появится как untracked
git add README.md
git status                                    # README.md теперь staged
git commit -m "Начальный коммит: добавлен README"

echo "print('Hello, Git!')" > main.py
git add .
git commit -m "Добавлен файл main.py"
```

### 7. Хорошие практики написания сообщений коммитов

- Используй настоящее время повелительного наклонения: «Добавить функцию», а не «Добавил функцию» (это распространённая, хотя и необязательная конвенция).
- Держи первую строку сообщения короткой (обычно до 50 символов) и по существу.
- Один коммит должен, по возможности, отражать одно логически законченное изменение, а не смешение множества несвязанных правок.

### Резюме урока

- `git init` создаёт новый репозиторий в текущей папке, добавляя служебную скрытую папку `.git`.
- Рабочий цикл Git: изменение файла → `git add` (добавление в индекс) → `git commit` (фиксация изменений с сообщением).
- `git status` — одна из самых часто используемых команд, показывающая текущее состояние отслеживаемых и неотслеживаемых файлов.

## Урок 1.3. Просмотр истории: git status, git log, git diff
**Время: 35 минут**

### Цель урока
Научиться просматривать историю изменений репозитория и сравнивать разные версии файлов.

### 1. Просмотр полной истории коммитов: git log

```bash
git log
```

Эта команда выводит список всех коммитов репозитория в обратном хронологическом порядке (от новых к старым), включая уникальный идентификатор коммита (хэш), автора, дату и сообщение.

### 2. Полезные флаги для git log

```bash
git log --oneline           # сокращённый вид: один коммит - одна строка
git log -3                     # показать только 3 последних коммита
git log --oneline --graph   # визуализация истории веток в виде графа (текстовая схема)
git log --author="Алина"    # фильтр коммитов по конкретному автору
```

### 3. Идентификатор коммита (хэш)

Каждый коммит в Git получает уникальный идентификатор — длинную шестнадцатеричную строку (хэш), например `a1b2c3d4...`. Обычно достаточно первых 6-7 символов хэша, чтобы однозначно сослаться на конкретный коммит в большинстве команд Git.

### 4. Просмотр изменений: git diff

`git diff` показывает построчные различия между разными состояниями файлов:

```bash
git diff                   # изменения в рабочей папке, ещё НЕ добавленные в индекс (git add)
git diff --staged          # изменения, уже добавленные в индекс, но ещё не закоммиченные
git diff HEAD~1 HEAD       # различия между предыдущим и последним коммитом
```

В выводе `git diff` строки, начинающиеся с `-`, означают удалённый текст, строки с `+` — добавленный.

### 5. Что такое HEAD

`HEAD` — специальная ссылка Git, всегда указывающая на текущий коммит (обычно последний коммит текущей активной ветки). `HEAD~1` означает «коммит перед текущим», `HEAD~2` — «на два коммита раньше текущего» и так далее.

### 6. Просмотр конкретного коммита: git show

```bash
git show a1b2c3d      # показывает подробную информацию о конкретном коммите по его хэшу (или части хэша)
```

### 7. Практический пример: работа с историей

```bash
git log --oneline
# a1b2c3d Добавлен файл main.py
# f4e5d6c Начальный коммит: добавлен README

git diff HEAD~1 HEAD    # что изменилось между этими двумя коммитами
git show a1b2c3d           # подробности конкретного коммита
```

### Резюме урока

- `git log` показывает историю всех коммитов, `git log --oneline` — в удобном сокращённом виде.
- `git diff` показывает построчные различия между версиями файлов на разных этапах (рабочая папка, индекс, коммиты).
- `HEAD` — специальная ссылка, всегда указывающая на текущий (обычно последний) коммит активной ветки.

# Модуль 2. Ветвление и работа с удалённым репозиторием

---

## Урок 2.1. Ветки: git branch, git checkout/switch
**Время: 45 минут**

### Цель урока
Освоить работу с ветками Git — ключевым механизмом, позволяющим вести параллельную разработку.

### 1. Зачем нужны ветки

Ветка (branch) — независимая линия разработки внутри одного репозитория. Ветки позволяют работать над новой функцией или исправлением ошибки, не затрагивая основной, стабильный код проекта, и затем безопасно объединить готовые изменения обратно, когда работа завершена и проверена.

### 2. Главная ветка

По умолчанию при создании репозитория (`git init`) создаётся главная ветка, обычно называемая `main` (в более ранних проектах часто встречается историческое название `master`) — она считается основной, «стабильной» веткой проекта.

### 3. Просмотр списка веток

```bash
git branch
```

Текущая активная ветка в выводе этой команды отмечается звёздочкой `*`.

### 4. Создание новой ветки

```bash
git branch feature-login   # создаёт новую ветку, но НЕ переключается на неё автоматически
```

### 5. Переключение между ветками: git checkout / git switch

```bash
git checkout feature-login    # переключение на существующую ветку (классический способ)
git switch feature-login        # переключение на существующую ветку (современный, более явный способ)
```

`git switch` — более новая команда (появилась в Git 2.23), созданная специально для переключения веток, тогда как `git checkout` исторически выполняет сразу несколько разных функций (переключение веток, восстановление файлов и др.), что иногда приводит к путанице.

### 6. Создание ветки и переключение на неё одной командой

```bash
git checkout -b feature-login   # классический способ: создать и сразу переключиться
git switch -c feature-login       # современный способ (флаг -c от "create")
```

### 7. Практический пример

```bash
git branch                          # проверяем список веток, видим только main
git switch -c feature-login       # создаём новую ветку и переключаемся на неё

echo "Код новой функции" >> login.py
git add login.py
git commit -m "Добавлена функция входа в систему"

git switch main                     # возвращаемся на основную ветку
git branch                          # теперь видим обе ветки: main и feature-login
```

### 8. Удаление ветки

```bash
git branch -d feature-login    # удалить ветку (безопасно - Git предупредит, если изменения не слиты)
git branch -D feature-login    # принудительное удаление, даже если изменения не были слиты
```

### Резюме урока

- Ветка — независимая линия разработки, позволяющая работать над изменениями, не затрагивая основной код проекта.
- `git switch` — современная, рекомендуемая команда для переключения между ветками, `git switch -c` создаёт новую ветку и сразу переключается на неё.
- Текущая активная ветка отмечается звёздочкой в выводе команды `git branch`.

## Урок 2.2. Слияние веток и разрешение конфликтов (merge)
**Время: 55 минут**

### Цель урока
Научиться объединять изменения из разных веток и разрешать возникающие конфликты слияния.

### 1. Слияние веток: git merge

После завершения работы в отдельной ветке её изменения объединяются с основной веткой командой `git merge`:

```bash
git switch main               # переключаемся на ветку, В КОТОРУЮ хотим влить изменения
git merge feature-login          # вливаем изменения из feature-login в текущую ветку (main)
```

### 2. Быстрое слияние (fast-forward)

Если с момента создания дочерней ветки в основной ветке не было сделано никаких новых коммитов, Git выполняет так называемое «быстрое слияние» (fast-forward) — просто перемещает указатель основной ветки вперёд, без создания отдельного «коммита слияния».

### 3. Слияние с созданием отдельного коммита (three-way merge)

Если в обеих ветках были сделаны независимые изменения с момента их расхождения, Git создаёт специальный «коммит слияния», объединяющий историю обеих веток. Такой коммит обычно имеет автоматически сгенерированное сообщение вида «Merge branch 'feature-login'».

### 4. Что такое конфликт слияния

Конфликт возникает, когда одни и те же строки одного и того же файла были изменены по-разному в двух сливаемых ветках — в этом случае Git не может автоматически определить, какой вариант правильный, и требует ручного вмешательства разработчика.

### 5. Как выглядит конфликт в файле

```
<<<<<<< HEAD
код из текущей ветки (в которую вливаем, например main)
=======
код из вливаемой ветки (например feature-login)
>>>>>>> feature-login
```

### 6. Пошаговое разрешение конфликта

1. Открыть файл с конфликтом в текстовом редакторе — Git явно помечает конфликтующие места указанными выше маркерами.
2. Вручную отредактировать файл, оставив нужный вариант кода (или объединив оба варианта осмысленным образом).
3. Полностью удалить служебные маркеры `<<<<<<<`, `=======`, `>>>>>>>`.
4. Добавить исправленный файл в индекс и завершить слияние коммитом:

```bash
git add file.txt
git commit -m "Разрешён конфликт слияния между main и feature-login"
```

### 7. Отмена незавершённого слияния

Если конфликт кажется слишком сложным для немедленного разрешения, слияние можно полностью отменить и вернуться к состоянию до его начала:

```bash
git merge --abort
```

### 8. Как минимизировать конфликты слияния

- Чаще синхронизировать свою рабочую ветку с изменениями из основной ветки (регулярно подтягивая новые коммиты).
- Стараться делать ветки короткоживущими — чем дольше ветка существует отдельно, тем выше риск существенных конфликтов при слиянии.
- По возможности разделять работу над проектом так, чтобы разные разработчики реже редактировали одни и те же участки одних и тех же файлов одновременно.

### Резюме урока

- `git merge` объединяет изменения из одной ветки в другую (обычно текущую), выполняя либо быстрое слияние, либо создавая отдельный коммит слияния.
- Конфликт слияния возникает при противоречащих изменениях одних и тех же строк файла и требует ручного разрешения через редактирование файла и удаление служебных маркеров.
- `git merge --abort` полностью отменяет начатое, но незавершённое слияние, возвращая репозиторий к состоянию до его начала.

## Урок 2.3. Работа с GitHub: клонирование, push, pull
**Время: 45 минут**

### Цель урока
Научиться синхронизировать локальный репозиторий с удалённым сервером (GitHub).

### 1. Что такое GitHub

GitHub — крупнейшая облачная платформа для хранения Git-репозиториев, предоставляющая, помимо самого хранения кода, инструменты для командной работы: Issues (задачи и баг-репорты), Pull Requests (запросы на слияние изменений, Урок 2.4), а также автоматизацию через GitHub Actions.

### 2. Создание репозитория на GitHub

1. Зарегистрироваться на сайте github.com.
2. Нажать кнопку «New repository», указать название и видимость репозитория (публичный или приватный).
3. GitHub автоматически предложит набор команд для подключения уже существующего локального репозитория либо для создания нового.

### 3. Подключение локального репозитория к GitHub: git remote

```bash
git remote add origin https://github.com/username/my_project.git
git remote -v      # проверка, какие удалённые репозитории подключены
```

`origin` — стандартное, общепринятое имя для основного удалённого репозитория проекта (технически можно назвать иначе, но это устоявшаяся конвенция).

### 4. Отправка изменений: git push

```bash
git push -u origin main    # первая отправка: флаг -u связывает локальную ветку main с origin/main
git push                      # все последующие отправки (после первой можно без дополнительных флагов)
```

### 5. Клонирование существующего репозитория: git clone

```bash
git clone https://github.com/username/my_project.git
```

Эта команда скачивает полную копию удалённого репозитория, включая всю его историю коммитов, на локальный компьютер, автоматически настраивая связь с `origin`.

### 6. Получение изменений: git pull

```bash
git pull origin main
```

`git pull` фактически представляет собой комбинацию из двух отдельных операций: `git fetch` (скачивание новых изменений с сервера без их слияния) и `git merge` (слияние скачанных изменений с локальной веткой).

### 7. Аутентификация при работе с GitHub

Начиная с 2021 года GitHub больше не поддерживает аутентификацию по обычному паролю через командную строку — вместо этого требуется либо Personal Access Token (персональный токен доступа, создаваемый в настройках аккаунта), либо настройка SSH-ключей для более удобной и безопасной работы без повторного ввода учётных данных.

### 8. Практический пример: полный цикл работы с удалённым репозиторием

```bash
git clone https://github.com/username/example.git
cd example

echo "новая строка" >> notes.txt
git add notes.txt
git commit -m "Обновлены заметки"
git push

git pull    # получение изменений, сделанных другими участниками проекта
```

### Резюме урока

- GitHub — облачная платформа для хранения Git-репозиториев и совместной работы над кодом с дополнительными инструментами (Issues, Pull Requests).
- `git push` отправляет локальные изменения на сервер, `git pull` получает и объединяет изменения с сервера.
- `git clone` создаёт полную локальную копию существующего удалённого репозитория, включая всю его историю.

## Урок 2.4. Совместная работа: Pull Request, Fork, .gitignore
**Время: 35 минут**

### Цель урока
Освоить стандартный рабочий процесс командной разработки через GitHub и научиться исключать ненужные файлы из репозитория.

### 1. Что такое Pull Request

Pull Request (PR) — запрос на слияние изменений из одной ветки в другую (обычно из отдельной feature-ветки в основную ветку `main`), сопровождающийся возможностью обсуждения предложенных изменений, проверки кода коллегами (code review) и автоматических проверок перед фактическим слиянием.

### 2. Типичный процесс работы через Pull Request

1. Создать новую ветку от актуальной версии `main` для работы над конкретной задачей.
2. Внести необходимые изменения, зафиксировав их через один или несколько коммитов.
3. Отправить ветку на GitHub через `git push`.
4. Создать Pull Request из этой ветки в `main` через интерфейс GitHub.
5. Дождаться проверки кода коллегами и/или прохождения автоматических тестов.
6. После получения одобрения — выполнить слияние (merge) через интерфейс GitHub.

### 3. Что такое Fork

Fork (форк) — создание собственной полной копии чужого репозитория в своём аккаунте GitHub. Форки особенно часто используются при работе с проектами с открытым исходным кодом (open source): вместо получения прямого доступа на запись к чужому репозиторию, разработчик форкает его, вносит изменения в своей копии, а затем предлагает эти изменения оригинальному автору через Pull Request из своего форка.

### 4. Разница между веткой и форком

Ветка (branch) существует внутри одного и того же репозитория и обычно используется при наличии прав на запись к нему (например, для членов одной команды). Форк создаёт полностью отдельную копию всего репозитория в другом аккаунте — обычно используется, когда у разработчика нет прямых прав записи в оригинальный репозиторий.

### 5. Файл .gitignore

Файл `.gitignore` указывает Git, какие файлы и папки не нужно отслеживать вообще — это критически важно перед началом совместной работы, чтобы избежать случайного попадания в репозиторий временных файлов, файлов с секретными данными или сгенерированных файлов сборки:

```
node_modules/
.env
*.log
__pycache__/
.DS_Store
```

Файл `.gitignore` должен быть создан и добавлен в репозиторий как можно раньше — файлы, уже однажды закоммиченные, не начнут автоматически игнорироваться только из-за добавления в `.gitignore` (для их удаления из отслеживания потребуются дополнительные команды).

### 6. Хорошие практики оформления Pull Request

Качественный Pull Request обычно включает: краткое, но информативное описание того, что было сделано и зачем; ссылку на связанную задачу (issue), если применимо; при необходимости — скриншоты или пошаговые инструкции, облегчающие проверку изменений коллегой.

### 7. Практический пример: цикл через Pull Request

```bash
git switch main
git pull origin main                     # убеждаемся, что main актуальна перед началом работы
git switch -c feature-user-profile     # создаём новую ветку под конкретную задачу

# ... вносим необходимые изменения в код ...

git add .
git commit -m "Добавлена страница профиля пользователя"
git push -u origin feature-user-profile
```

После выполнения этих команд на GitHub автоматически появится предложение создать Pull Request для только что отправленной ветки.

### Резюме урока

- Pull Request — стандартный механизм GitHub для предложения, обсуждения и проверки изменений перед их слиянием в основную ветку проекта.
- Форк создаёт полную отдельную копию репозитория в другом аккаунте — используется, когда нет прямых прав на запись в оригинальный репозиторий, в отличие от обычной ветки.
- Файл `.gitignore` необходимо настроить как можно раньше, чтобы избежать случайного попадания временных файлов и секретных данных в историю репозитория.
