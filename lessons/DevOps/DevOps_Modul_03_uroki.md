# Модуль 3. Системы контроля версий (Git/GitHub)

---

## Урок 3.1. Зачем нужен Git, установка и настройка
**Время: 35 минут**

### Цель урока
Понять назначение систем контроля версий, установить и настроить Git для дальнейшей работы.

### 1. Проблема, которую решает Git

Без системы контроля версий разработчики часто хранят историю изменений вручную — через папки вида `project_final`, `project_final_v2`, `project_final_REAL_v3`. Это неудобно, ненадёжно и делает практически невозможной совместную работу нескольких человек над одним кодом.

Git — это распределённая система контроля версий, которая отслеживает все изменения в файлах проекта, позволяет вернуться к любой предыдущей версии, а также даёт возможность нескольким разработчикам параллельно работать над одним и тем же проектом без потери изменений.

### 2. Почему Git — стандарт индустрии

- Позволяет видеть полную историю изменений: кто, когда и что именно изменил.
- Поддерживает параллельную разработку через ветки (branches) без риска потерять чужие изменения.
- Работает локально — большинство операций не требует подключения к интернету.
- Является основой большинства современных DevOps-практик: CI/CD пайплайны запускаются именно по событиям в Git-репозитории.

### 3. Установка Git

**Windows:** скачать установщик с сайта git-scm.com и установить с настройками по умолчанию.

**macOS:**
```bash
brew install git
```

**Linux (Ubuntu/Debian):**
```bash
sudo apt update
sudo apt install git
```

Проверка установки:
```bash
git --version
```

### 4. Первоначальная настройка Git

После установки необходимо указать своё имя и email — эта информация будет прикрепляться к каждому изменению (коммиту):

```bash
git config --global user.name "Алина Иванова"
git config --global user.email "alina@example.com"
```

Флаг `--global` означает, что настройки применяются ко всем репозиториям на этом компьютере.

### 5. Проверка текущих настроек

```bash
git config --list
```

### 6. Основная терминология Git

| Термин | Значение |
|---|---|
| Репозиторий (repository) | папка проекта, отслеживаемая Git |
| Коммит (commit) | сохранённая «точка» в истории изменений |
| Ветка (branch) | независимая линия разработки |
| Remote (удалённый репозиторий) | копия репозитория на сервере (например, GitHub) |

### Резюме урока

- Git — распределённая система контроля версий, отслеживающая изменения в файлах проекта и позволяющая нескольким разработчикам работать параллельно.
- После установки Git необходимо настроить имя и email через `git config --global`.
- Git лежит в основе большинства современных DevOps-практик, включая автоматический запуск CI/CD пайплайнов.

## Урок 3.2. Базовые команды: init, add, commit, status, log
**Время: 45 минут**

### Цель урока
Освоить базовый рабочий цикл Git: создание репозитория и сохранение изменений.

### 1. Создание нового репозитория: git init

```bash
mkdir my_project
cd my_project
git init
```

Команда `git init` создаёт в текущей папке скрытую папку `.git`, где Git хранит всю историю изменений и служебную информацию.

### 2. Три состояния файлов в Git

- **Untracked (неотслеживаемый)** — файл существует в папке, но Git ещё не отслеживает его изменения.
- **Staged (проиндексированный)** — файл помечен для включения в следующий коммит.
- **Committed (зафиксированный)** — изменения файла сохранены в истории.

### 3. Проверка состояния: git status

```bash
git status
```

Эта команда — одна из самых часто используемых в Git, она показывает, какие файлы изменены, добавлены в индекс или ещё не отслеживаются.

### 4. Добавление файлов в индекс: git add

```bash
git add file.txt          # добавить конкретный файл
git add .                  # добавить все изменённые файлы в текущей папке
```

### 5. Сохранение изменений: git commit

```bash
git commit -m "Добавлен файл конфигурации"
```

Флаг `-m` позволяет указать сообщение коммита прямо в команде. Хорошее сообщение коммита кратко и понятно объясняет, что именно было изменено и зачем.

### 6. Просмотр истории: git log

```bash
git log                 # полная история коммитов
git log --oneline       # сокращённый вид: одна строка на коммит
git log -3               # последние 3 коммита
```

### 7. Практический пример: полный рабочий цикл

```bash
mkdir demo_project
cd demo_project
git init

echo "# Мой проект" > README.md
git add README.md
git commit -m "Начальный коммит: добавлен README"

echo "print('Hello')" > main.py
git add .
git commit -m "Добавлен файл main.py"

git log --oneline
```

### Резюме урока

- `git init` создаёт новый репозиторий, `git status` показывает текущее состояние отслеживаемых файлов.
- Рабочий цикл Git: изменение файла → `git add` (индексация) → `git commit` (фиксация изменений).
- `git log` показывает историю всех коммитов в репозитории.

## Урок 3.3. Ветвление и слияние (branch, merge, конфликты)
**Время: 50 минут**

### Цель урока
Освоить работу с ветками Git — ключевым механизмом параллельной разработки — и научиться разрешать конфликты слияния.

### 1. Зачем нужны ветки

Ветка (branch) — это независимая линия разработки внутри репозитория. Ветки позволяют работать над новой функцией или исправлением ошибки, не затрагивая основной («стабильный») код, и затем объединить изменения обратно, когда работа завершена и протестирована.

### 2. Главная ветка

По умолчанию при создании репозитория создаётся главная ветка, обычно называемая `main` (ранее часто называлась `master`) — она считается основной, «стабильной» веткой проекта.

### 3. Создание и переключение веток

```bash
git branch feature-login          # создать новую ветку
git checkout feature-login          # переключиться на неё

# современный сокращённый вариант обеих команд сразу:
git switch -c feature-login
```

### 4. Просмотр списка веток

```bash
git branch              # список локальных веток, текущая отмечена звёздочкой
```

### 5. Слияние веток: git merge

После завершения работы в отдельной ветке её изменения объединяются с основной веткой:

```bash
git checkout main               # переключаемся на основную ветку
git merge feature-login            # вливаем изменения из feature-login в main
```

### 6. Конфликты слияния

Конфликт возникает, когда одни и те же строки файла были изменены по-разному в двух сливаемых ветках — Git не может автоматически решить, какой вариант правильный, и требует ручного вмешательства:

```
<<<<<<< HEAD
код из текущей ветки (main)
=======
код из вливаемой ветки (feature-login)
>>>>>>> feature-login
```

Для разрешения конфликта нужно вручную отредактировать файл, оставив нужный вариант (или объединив оба), удалить служебные маркеры `<<<<<<<`, `=======`, `>>>>>>>`, а затем зафиксировать результат:

```bash
git add file.txt
git commit -m "Разрешён конфликт слияния"
```

### 7. Удаление ветки после слияния

```bash
git branch -d feature-login       # удалить локальную ветку после успешного слияния
```

### Резюме урока

- Ветки позволяют параллельно вести независимую разработку, не затрагивая основной код проекта.
- `git branch`/`git switch -c` создают и переключают ветки, `git merge` объединяет изменения обратно.
- Конфликты слияния возникают при противоречащих изменениях одних и тех же строк и требуют ручного разрешения.

## Урок 3.4. Работа с удалённым репозиторием: GitHub, push/pull
**Время: 45 минут**

### Цель урока
Научиться синхронизировать локальный репозиторий с удалённым сервером (GitHub) для резервного копирования и совместной работы.

### 1. Что такое GitHub

GitHub — облачная платформа для хранения Git-репозиториев и совместной работы над кодом. Помимо самого хранения кода, GitHub предоставляет инструменты для командной работы: Issues (задачи), Pull Requests (запросы на слияние изменений), Actions (CI/CD, подробнее в Модуле 5).

### 2. Создание репозитория на GitHub

1. Зарегистрируйся на github.com.
2. Нажми «New repository», укажи название и видимость (публичный/приватный).
3. GitHub предложит команды для подключения существующего локального репозитория или создания нового.

### 3. Подключение локального репозитория к GitHub: git remote

```bash
git remote add origin https://github.com/username/my_project.git
git remote -v            # проверка подключённых удалённых репозиториев
```

`origin` — стандартное имя для основного удалённого репозитория (можно назвать иначе, но это общепринятая конвенция).

### 4. Отправка изменений: git push

```bash
git push -u origin main    # первая отправка, флаг -u привязывает локальную ветку к удалённой
git push                     # последующие отправки (после первой)
```

### 5. Клонирование существующего репозитория: git clone

```bash
git clone https://github.com/username/my_project.git
```

Эта команда скачивает полную копию репозитория вместе со всей историей изменений на локальный компьютер.

### 6. Получение изменений: git pull

```bash
git pull origin main
```

`git pull` — это, по сути, комбинация двух операций: `git fetch` (скачивание изменений с сервера) и `git merge` (слияние их с локальной веткой).

### 7. Аутентификация при работе с GitHub

Начиная с 2021 года GitHub больше не поддерживает вход по паролю через командную строку — вместо этого используется Personal Access Token (персональный токен доступа) или SSH-ключи, которые нужно предварительно настроить в личных настройках аккаунта GitHub.

### Резюме урока

- GitHub — платформа для хранения Git-репозиториев и совместной работы над кодом в облаке.
- `git push` отправляет локальные изменения на сервер, `git pull` — забирает изменения с сервера.
- `git clone` создаёт полную локальную копию существующего удалённого репозитория.

## Урок 3.5. Практика: совместная работа через Pull Request
**Время: 45 минут**

### Цель урока
Освоить стандартный рабочий процесс командной разработки через GitHub с использованием Pull Request.

### 1. Что такое Pull Request (PR)

Pull Request — это запрос на слияние изменений из одной ветки (обычно feature-ветки) в другую (обычно `main`), сопровождающийся возможностью обсуждения, проверки кода (code review) и автоматических проверок (CI) перед фактическим слиянием.

### 2. Типичный рабочий процесс (Git Flow, упрощённый)

1. Создать новую ветку от актуальной `main` для работы над задачей.
2. Внести изменения, зафиксировать их через коммиты.
3. Отправить ветку на GitHub через `git push`.
4. Создать Pull Request из этой ветки в `main`.
5. Дождаться проверки кода коллегами и/или прохождения автоматических тестов.
6. После одобрения — слияние (merge) через интерфейс GitHub.

### 3. Пошаговый пример

```bash
git checkout main
git pull origin main                  # убеждаемся, что main актуальна
git switch -c feature-user-profile    # создаём новую ветку

# ... вносим изменения в код ...

git add .
git commit -m "Добавлена страница профиля пользователя"
git push -u origin feature-user-profile
```

После этого на GitHub появится предложение создать Pull Request для только что отправленной ветки.

### 4. Что писать в описании Pull Request

Хороший Pull Request содержит: краткое описание, что было сделано и зачем; ссылку на связанную задачу (issue), если применимо; при необходимости — скриншоты или инструкции для проверяющего.

### 5. Code review (проверка кода)

Проверяющий коллега может оставлять комментарии к конкретным строкам кода прямо в интерфейсе GitHub, запрашивать изменения или одобрять Pull Request. После получения комментариев автор вносит правки, коммитит их в ту же ветку — они автоматически появляются в том же Pull Request.

### 6. Файл .gitignore

Перед началом совместной работы важно настроить файл `.gitignore`, указывающий Git, какие файлы и папки не нужно отслеживать (временные файлы, файлы с секретными данными, зависимости):

```
node_modules/
.env
*.log
__pycache__/
```

### Резюме урока

- Pull Request — стандартный механизм GitHub для предложения, обсуждения и проверки изменений перед их слиянием в основную ветку.
- Типичный процесс: новая ветка → изменения → push → Pull Request → code review → merge.
- Файл `.gitignore` предотвращает попадание временных файлов и секретных данных в репозиторий — важная практика перед началом совместной работы.
