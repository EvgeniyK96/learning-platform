# Модуль 6. Облачные платформы и виртуализация

---

## Урок 6.1. Что такое облако: IaaS, PaaS, SaaS
**Время: 35 минут**

### Цель урока
Понять базовые концепции облачных вычислений и разобраться в основных моделях облачных услуг.

### 1. Что такое облачные вычисления

Облачные вычисления — это предоставление вычислительных ресурсов (серверов, хранилищ данных, баз данных, сетей) через интернет по модели «оплата по мере использования», без необходимости покупать и обслуживать собственное физическое оборудование.

### 2. Почему компании переходят в облако

- **Экономия на инфраструктуре.** Не нужно закупать дорогое оборудование заранее — платишь только за фактически используемые ресурсы.
- **Масштабируемость.** Ресурсы можно быстро увеличить при росте нагрузки и уменьшить при её снижении.
- **Глобальная доступность.** Облачные провайдеры имеют дата-центры по всему миру, что позволяет размещать приложения ближе к пользователям.
- **Снижение нагрузки на команду.** Провайдер берёт на себя обслуживание физического оборудования, электропитания, охлаждения.

### 3. Три основные модели облачных услуг

- **IaaS (Infrastructure as a Service)** — предоставление базовой инфраструктуры: виртуальных серверов, хранилищ, сетей. Пользователь сам устанавливает и настраивает операционную систему и ПО. Пример: Amazon EC2, Google Compute Engine.
- **PaaS (Platform as a Service)** — предоставление платформы для разработки и запуска приложений без необходимости управлять серверами напрямую. Пример: Heroku, Google App Engine.
- **SaaS (Software as a Service)** — готовое к использованию программное обеспечение, доступное через интернет. Пример: Gmail, Slack, Zoom.

### 4. Наглядное сравнение уровня ответственности

| | On-premise (свои серверы) | IaaS | PaaS | SaaS |
|---|---|---|---|---|
| Приложение | Вы | Вы | Вы | Провайдер |
| Данные и настройки | Вы | Вы | Вы | Вы (частично) |
| Операционная система | Вы | Вы | Провайдер | Провайдер |
| Физическое оборудование | Вы | Провайдер | Провайдер | Провайдер |

Чем выше уровень абстракции (от IaaS к SaaS), тем меньше контроля, но и меньше операционной нагрузки на пользователя.

### 5. Где здесь место DevOps-инженера

DevOps-инженеры чаще всего работают на уровне IaaS и PaaS: разворачивают и настраивают виртуальные серверы, конфигурируют сети, автоматизируют управление инфраструктурой через код (Модуль 8), при этом используя готовые управляемые сервисы (базы данных, очереди сообщений) там, где это уместно, чтобы не тратить время на их самостоятельную настройку и поддержку.

### 6. Публичное, частное и гибридное облако

- **Публичное облако** — ресурсы провайдера используются совместно множеством клиентов (AWS, GCP, Azure).
- **Частное облако** — инфраструктура выделена для одной организации, часто размещается в собственном дата-центре компании.
- **Гибридное облако** — сочетание собственной инфраструктуры и публичного облака, например для соблюдения требований к хранению определённых данных.

### Резюме урока

- Облачные вычисления предоставляют вычислительные ресурсы по модели «оплата по мере использования» через интернет.
- Три модели облачных услуг: IaaS (инфраструктура), PaaS (платформа), SaaS (готовое ПО) — отличаются уровнем контроля и операционной нагрузки на пользователя.
- DevOps-инженеры чаще всего работают на уровне IaaS и PaaS, комбинируя их с готовыми управляемыми сервисами.

## Урок 6.2. Обзор основных провайдеров (AWS, GCP, Azure)
**Время: 40 минут**

### Цель урока
Получить общее представление о трёх крупнейших облачных провайдерах и их основных сервисах.

### 1. Amazon Web Services (AWS)

AWS — крупнейший облачный провайдер в мире, предлагающий сотни различных сервисов. Ключевые из них для DevOps:

- **EC2 (Elastic Compute Cloud)** — виртуальные серверы.
- **S3 (Simple Storage Service)** — объектное хранилище файлов.
- **RDS (Relational Database Service)** — управляемые реляционные базы данных.
- **Lambda** — выполнение кода без управления серверами (serverless).
- **ECS/EKS** — управление контейнерами и Kubernetes.
- **IAM (Identity and Access Management)** — управление правами доступа.

### 2. Google Cloud Platform (GCP)

GCP отличается сильными возможностями в области анализа данных и машинного обучения, а также одним из лучших управляемых Kubernetes-сервисов на рынке.

- **Compute Engine** — виртуальные серверы.
- **Cloud Storage** — объектное хранилище.
- **Cloud SQL** — управляемые базы данных.
- **Google Kubernetes Engine (GKE)** — управляемый Kubernetes, один из самых зрелых сервисов такого рода.
- **Cloud Functions** — serverless-вычисления.

### 3. Microsoft Azure

Azure особенно популярен в компаниях, уже использующих экосистему Microsoft (Windows Server, Active Directory, .NET).

- **Virtual Machines** — виртуальные серверы.
- **Blob Storage** — объектное хранилище.
- **Azure SQL Database** — управляемые базы данных.
- **Azure Kubernetes Service (AKS)** — управляемый Kubernetes.
- **Azure Functions** — serverless-вычисления.

### 4. Сравнение основных сервисов между провайдерами

| Назначение | AWS | GCP | Azure |
|---|---|---|---|
| Виртуальные серверы | EC2 | Compute Engine | Virtual Machines |
| Объектное хранилище | S3 | Cloud Storage | Blob Storage |
| Управляемый Kubernetes | EKS | GKE | AKS |
| Serverless-функции | Lambda | Cloud Functions | Azure Functions |

### 5. Какого провайдера выбрать для изучения

Все три провайдера решают похожие задачи схожими по концепции сервисами — освоив глубоко один из них, значительно проще разобраться в остальных, так как базовые концепции (виртуальные серверы, сети, хранилища, управление доступом) универсальны. AWS остаётся наиболее распространённым выбором на рынке труда, поэтому часто рекомендуется в качестве первого облака для изучения, но выбор конкретного провайдера часто определяется требованиями компании или проекта.

### 6. Бесплатные уровни для практики

У большинства крупных провайдеров есть бесплатный уровень (free tier) с ограниченными по времени или объёму ресурсами — этого обычно достаточно для учебных экспериментов без реальных затрат, но важно внимательно следить за использованием ресурсов, чтобы не выйти за границы бесплатного лимита.

### Резюме урока

- AWS, GCP и Azure — три крупнейших облачных провайдера, предлагающих концептуально схожие сервисы под разными названиями.
- AWS — наиболее распространён на рынке труда, GCP — силён в Kubernetes и данных, Azure — популярен в экосистеме Microsoft.
- Освоение базовых концепций (виртуальные серверы, хранилища, сети) на одном провайдере значительно облегчает переход на любой другой.

## Урок 6.3. Виртуальные машины vs контейнеры
**Время: 35 минут**

### Цель урока
Углубить понимание различий между виртуальными машинами и контейнерами и определить, когда что уместнее использовать.

### 1. Повторение основ (см. также Урок 4.1)

Виртуальная машина эмулирует полную операционную систему поверх гипервизора, контейнер — использует общее ядро ОС хоста, изолируя только уровень приложения.

### 2. Что такое гипервизор

Гипервизор — программное обеспечение, позволяющее запускать несколько виртуальных машин на одном физическом сервере, распределяя между ними ресурсы (CPU, память, диск). Примеры: VMware, VirtualBox, Hyper-V, а также облачные гипервизоры провайдеров (AWS Nitro и т.д.).

### 3. Когда предпочтительнее виртуальные машины

- Требуется полная изоляция на уровне операционной системы (например, для строгих требований безопасности).
- Приложениям на одном сервере нужны разные ядра ОС (например, одновременно Windows и Linux).
- Требуется запускать legacy-приложения, плохо адаптированные для контейнеризации.

### 4. Когда предпочтительнее контейнеры

- Нужен быстрый запуск и остановка (секунды, а не минуты).
- Приложение состоит из множества небольших независимых сервисов (микросервисная архитектура).
- Важна максимальная эффективность использования ресурсов сервера (контейнеры значительно легче ВМ).
- Требуется обеспечить идентичность окружения разработки и продакшена.

### 5. Совместное использование в реальной практике

На практике виртуальные машины и контейнеры часто используются вместе, а не как взаимоисключающие технологии: облачный провайдер предоставляет виртуальные машины (уровень IaaS), а на этих виртуальных машинах уже запускаются Docker-контейнеры (или кластер Kubernetes, Модуль 7) с приложениями.

### 6. Serverless как следующий уровень абстракции

Помимо виртуальных машин и контейнеров, существует ещё более высокий уровень абстракции — serverless-вычисления (например, AWS Lambda): разработчик пишет только код функции, а провайдер полностью берёт на себя управление серверами, масштабированием и инфраструктурой. За такое удобство приходится платить меньшей гибкостью в настройке окружения.

### Резюме урока

- Виртуальные машины обеспечивают более сильную изоляцию за счёт эмуляции полной ОС, контейнеры — более лёгкие и быстрые за счёт использования общего ядра ОС хоста.
- Выбор между ВМ и контейнерами зависит от требований к изоляции, скорости запуска и архитектуры приложения.
- В реальной практике эти технологии часто комбинируются: контейнеры запускаются поверх облачных виртуальных машин.

## Урок 6.4. Развёртывание простого сервера в облаке
**Время: 55 минут**

### Цель урока
На практике развернуть простой веб-сервер на облачной виртуальной машине.

### 1. Общий процесс развёртывания сервера в облаке

Хотя интерфейсы у разных провайдеров отличаются, общий процесс схож:
1. Создать аккаунт у облачного провайдера.
2. Создать виртуальную машину (инстанс), выбрав образ ОС (например, Ubuntu), размер (CPU, память) и регион дата-центра.
3. Настроить правила сети (firewall) — какие порты открыты для доступа извне.
4. Подключиться к серверу по SSH.
5. Установить и настроить необходимое ПО.

### 2. Создание SSH-ключа

Для безопасного подключения к облачному серверу используется пара SSH-ключей (приватный и публичный) вместо пароля:

```bash
ssh-keygen -t ed25519 -C "my-cloud-server"
```

Публичный ключ добавляется в настройки виртуальной машины при её создании, приватный — остаётся на локальном компьютере и используется для подключения.

### 3. Подключение к серверу по SSH

```bash
ssh -i ~/.ssh/my_key ubuntu@<IP_адрес_сервера>
```

### 4. Настройка правил сети (Security Group / Firewall)

При создании виртуальной машины важно явно разрешить входящий трафик по нужным портам — например, порт 22 для SSH-подключения и порт 80/443 для HTTP/HTTPS-трафика веб-сервера. По умолчанию большинство портов закрыты для внешнего доступа в целях безопасности.

### 5. Установка и запуск веб-сервера

После подключения по SSH можно установить и настроить простой веб-сервер, например, через Docker (Модуль 4):

```bash
sudo apt update
sudo apt install docker.io -y
sudo systemctl start docker

sudo docker run -d -p 80:80 nginx
```

После этого сайт должен быть доступен по публичному IP-адресу сервера в браузере.

### 6. Основы безопасности при работе с облачным сервером

- Никогда не используй пароль для SSH-подключения — только ключи.
- Ограничивай доступ по SSH только своим IP-адресом, где это возможно, а не «всему интернету» (0.0.0.0/0).
- Регулярно обновляй систему: `sudo apt update && sudo apt upgrade`.
- Не используй root-пользователя для повседневной работы — создавай отдельного пользователя с sudo-правами.

### Резюме урока

- Развёртывание сервера в облаке включает создание виртуальной машины, настройку сети и подключение по SSH.
- SSH-ключи — стандартный и более безопасный способ подключения к облачным серверам по сравнению с паролями.
- Правильная настройка правил сети (открытые только нужные порты) — базовая мера безопасности при работе с облачной инфраструктурой.

## Урок 6.5. Основы сетевой безопасности в облаке (firewall, ключи доступа)
**Время: 35 минут**

### Цель урока
Изучить базовые принципы обеспечения безопасности облачной инфраструктуры.

### 1. Принцип наименьших привилегий (Principle of Least Privilege)

Ключевой принцип безопасности: каждому пользователю, сервису или системе должны быть выданы только те права доступа, которые действительно необходимы для выполнения их задач — не больше. Это минимизирует потенциальный ущерб в случае компрометации учётных данных.

### 2. Firewall и Security Groups

Firewall (межсетевой экран) контролирует, какой сетевой трафик разрешён к серверу, а какой блокируется. В облачных провайдерах эта функциональность часто называется Security Groups (AWS) или Network Security Groups (Azure) и настраивается на уровне отдельных портов и протоколов:

```
Разрешить: TCP 22 (SSH) только с IP 203.0.113.5
Разрешить: TCP 443 (HTTPS) с любого адреса (0.0.0.0/0)
Запретить: всё остальное
```

### 3. Управление доступом: IAM

IAM (Identity and Access Management) — система управления пользователями, ролями и правами доступа в облачной платформе. Вместо предоставления полного административного доступа каждому сотруднику, создаются роли с ограниченным набором конкретных разрешений (например, «может читать данные из S3, но не может их удалять»).

### 4. Управление секретами (Secrets Management)

Пароли, API-ключи и прочие конфиденциальные данные никогда не должны храниться напрямую в коде или конфигурационных файлах репозитория. Вместо этого используются специализированные сервисы хранения секретов (AWS Secrets Manager, HashiCorp Vault) или, как минимум, переменные окружения и защищённые хранилища секретов CI/CD систем (см. Урок 5.5).

### 5. Шифрование данных

- **Шифрование в состоянии покоя (at rest)** — данные на диске хранятся в зашифрованном виде.
- **Шифрование при передаче (in transit)** — данные, передаваемые по сети, шифруются (HTTPS/TLS, а не HTTP).

Большинство облачных провайдеров предоставляют встроенные инструменты для обоих типов шифрования, важно не забывать их включать.

### 6. Регулярный аудит и обновления

Безопасность — не разовая настройка, а непрерывный процесс: регулярное обновление ПО и операционных систем, периодическая проверка списка предоставленных прав доступа (удаление неиспользуемых), мониторинг подозрительной активности.

### Резюме урока

- Принцип наименьших привилегий — базовая практика ограничения прав доступа только необходимым минимумом.
- Firewall/Security Groups контролируют, какой сетевой трафик разрешён к серверу, IAM — управляет правами пользователей и сервисов.
- Конфиденциальные данные (пароли, ключи) никогда не должны храниться напрямую в коде — для этого существуют специализированные системы управления секретами.
