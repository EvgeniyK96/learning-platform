# Модуль 9. Мониторинг, логирование и безопасность

---

## Урок 9.1. Зачем нужен мониторинг: метрики, алерты
**Время: 35 минут**

### Цель урока
Понять, зачем нужен мониторинг работающих систем, и изучить базовые понятия: метрики, алерты, дашборды.

### 1. Зачем нужен мониторинг

Мониторинг — это непрерывное отслеживание состояния работающей системы. Без мониторинга команда узнаёт о проблемах в приложении только от недовольных пользователей — что происходит уже после того, как проблема нанесла реальный ущерб. Хороший мониторинг позволяет обнаружить и устранить проблему до того, как её заметят пользователи.

### 2. Что такое метрика

Метрика — это числовое значение, отражающее состояние системы в определённый момент времени, собираемое регулярно (например, каждые несколько секунд). Примеры метрик:

- Загрузка CPU и использование памяти сервера.
- Количество запросов в секунду к веб-приложению.
- Среднее время ответа сервера (latency).
- Процент запросов, завершившихся ошибкой (error rate).
- Количество активных подключений к базе данных.

### 3. Четыре золотых сигнала (Four Golden Signals)

Популярный набор ключевых метрик, рекомендуемый для мониторинга практически любого сервиса (методология из книги Google о SRE):

- **Latency (задержка)** — сколько времени занимает обработка запроса.
- **Traffic (трафик)** — сколько запросов обрабатывает система.
- **Errors (ошибки)** — какая доля запросов завершается ошибкой.
- **Saturation (насыщение)** — насколько близко система к пределу своих возможностей (например, использование памяти близко к 100%).

### 4. Что такое алерт (alert)

Алерт — автоматическое уведомление, срабатывающее при выполнении определённого условия (например, «загрузка CPU превысила 90% в течение 5 минут» или «процент ошибок превысил 5%»). Алерты отправляются ответственной команде через email, Slack, SMS или специализированные сервисы (PagerDuty, Opsgenie).

### 5. Дашборд (dashboard)

Дашборд — визуальная панель, на которой в реальном времени отображаются ключевые метрики системы в виде графиков и индикаторов, позволяющая быстро оценить общее состояние системы «одним взглядом».

### 6. Проактивный vs реактивный подход

- **Реактивный мониторинг** — узнаём о проблеме, когда она уже произошла (алерт сработал, пользователи жалуются).
- **Проактивный мониторинг** — отслеживание трендов метрик позволяет предсказать проблему до её наступления (например, диск заполняется с определённой скоростью и закончится через 3 дня — можно среагировать заранее).

Хорошая система мониторинга поддерживает оба подхода.

### Резюме урока

- Мониторинг позволяет обнаруживать проблемы в работающей системе до того, как их заметят пользователи.
- Метрики — числовые показатели состояния системы, «Четыре золотых сигнала» (задержка, трафик, ошибки, насыщение) — стандартный набор для отслеживания.
- Алерты автоматически уведомляют команду при нарушении заданных условий, дашборды визуализируют состояние системы в реальном времени.

## Урок 9.2. Обзор инструментов: Prometheus, Grafana
**Время: 45 минут**

### Цель урока
Познакомиться с наиболее распространённой связкой инструментов мониторинга в DevOps-среде.

### 1. Что такое Prometheus

Prometheus — система мониторинга с открытым исходным кодом, специализирующаяся на сборе и хранении числовых метрик во времени (time-series данные). Prometheus периодически опрашивает («scrape») настроенные источники метрик и сохраняет собранные данные в собственной базе данных временных рядов.

### 2. Модель pull vs push

Prometheus использует модель «pull» — сам Prometheus с определённым интервалом запрашивает метрики у приложений (через специальный HTTP-эндпоинт `/metrics`), а не приложения самостоятельно отправляют («push») метрики в систему мониторинга. Такой подход упрощает конфигурацию и позволяет Prometheus контролировать частоту сбора данных.

### 3. Пример эндпоинта метрик

Приложения экспортируют метрики в специальном текстовом формате, понятном Prometheus:

```
# HELP http_requests_total Общее количество HTTP-запросов
# TYPE http_requests_total counter
http_requests_total{method="GET",status="200"} 1547
http_requests_total{method="POST",status="500"} 12
```

Для большинства популярных языков программирования и фреймворков существуют готовые библиотеки, автоматически экспортирующие подобные метрики.

### 4. PromQL — язык запросов Prometheus

Prometheus предоставляет собственный язык запросов (PromQL) для анализа и агрегации собранных метрик:

```promql
rate(http_requests_total[5m])
```

Этот запрос вычисляет среднюю скорость поступления запросов за последние 5 минут.

### 5. Что такое Grafana

Grafana — платформа для визуализации данных, часто используемая в связке с Prometheus (хотя поддерживает и множество других источников данных). Grafana позволяет создавать наглядные дашборды с графиками на основе данных, собранных Prometheus.

### 6. Типичная архитектура связки Prometheus + Grafana

```
Приложение (экспортирует метрики) 
        │
        ▼
   Prometheus (собирает и хранит метрики)
        │
        ▼
     Grafana (визуализирует метрики на дашбордах)
        │
        ▼
Alertmanager (отправляет уведомления при срабатывании алертов)
```

### 7. Готовые дашборды сообщества

Одно из больших преимуществ связки Prometheus + Grafana — обширное сообщество, публикующее готовые дашборды для популярных технологий (базы данных, веб-серверы, Kubernetes) — часто не нужно создавать визуализации с нуля, а можно импортировать готовый дашборд и адаптировать под свои нужды.

### Резюме урока

- Prometheus — система сбора и хранения числовых метрик, использующая модель «pull» для получения данных от приложений.
- PromQL — язык запросов для анализа собранных Prometheus метрик.
- Grafana визуализирует данные из Prometheus (и других источников) в виде наглядных дашбордов, часто с использованием готовых шаблонов сообщества.

## Урок 9.3. Централизованное логирование (ELK-стек, обзор)
**Время: 40 минут**

### Цель урока
Понять важность централизованного логирования и получить общее представление о популярном стеке инструментов ELK.

### 1. Проблема разрозненных логов

В распределённой системе, состоящей из десятков серверов и контейнеров, логи каждого компонента по умолчанию хранятся локально на своей машине. При расследовании проблемы разработчику пришлось бы вручную проверять логи на каждом сервере по отдельности — это медленно и неудобно, особенно если проблема затрагивает сразу несколько компонентов системы.

### 2. Что такое централизованное логирование

Централизованное логирование — практика сбора логов со всех компонентов системы в единое хранилище, где их можно искать, фильтровать и анализировать через единый интерфейс, вне зависимости от того, на каком именно сервере или в каком контейнере они были сгенерированы.

### 3. Что такое ELK-стек

ELK — популярный набор инструментов с открытым исходным кодом для централизованного логирования, аббревиатура от названий трёх основных компонентов:

- **Elasticsearch** — поисковая база данных, хранящая и индексирующая логи для быстрого поиска.
- **Logstash** — инструмент для сбора, обработки и преобразования логов перед отправкой в Elasticsearch.
- **Kibana** — веб-интерфейс для визуализации и поиска по логам, хранящимся в Elasticsearch.

Часто к этому набору добавляют **Filebeat** — лёгкий агент, устанавливаемый на каждом сервере/контейнере для отправки логов в Logstash или напрямую в Elasticsearch (в таком расширенном виде стек иногда называют «Elastic Stack»).

### 4. Типичный путь записи лога

```
Приложение (пишет лог)
     │
     ▼
  Filebeat (собирает и отправляет лог)
     │
     ▼
  Logstash (обрабатывает, фильтрует, обогащает лог)
     │
     ▼
Elasticsearch (индексирует и хранит лог)
     │
     ▼
   Kibana (поиск и визуализация в браузере)
```

### 5. Структурированное логирование

Для эффективной работы с централизованным логированием рекомендуется писать логи в структурированном формате (например, JSON), а не в виде произвольного текста:

```json
{"timestamp": "2026-07-12T10:00:00Z", "level": "ERROR", "service": "payment-api", "message": "Не удалось обработать платёж", "user_id": 12345}
```

Такой формат позволяет легко фильтровать и искать логи по конкретным полям (например, найти все ошибки конкретного сервиса за последний час).

### 6. Альтернативы ELK

Помимо ELK, существуют и другие популярные решения централизованного логирования: облачные сервисы провайдеров (AWS CloudWatch Logs, Google Cloud Logging), а также более лёгкие open-source альтернативы, такие как Grafana Loki, часто используемая в связке с уже знакомой Grafana (Урок 9.2) для единого интерфейса метрик и логов.

### Резюме урока

- Централизованное логирование собирает логи со всех компонентов распределённой системы в единое хранилище для удобного поиска и анализа.
- ELK-стек (Elasticsearch, Logstash, Kibana) — один из самых распространённых наборов инструментов для этой задачи.
- Структурированное логирование (например, в формате JSON) значительно упрощает последующий поиск и фильтрацию логов.

## Урок 9.4. Базовые принципы DevSecOps
**Время: 35 минут**

### Цель урока
Понять концепцию DevSecOps — интеграции практик безопасности в DevOps-процессы.

### 1. Что такое DevSecOps

DevSecOps — расширение философии DevOps, при котором вопросы безопасности («Security») встраиваются в процесс разработки и доставки ПО с самого начала, а не добавляются как отдельный финальный этап перед релизом. Принцип часто формулируется как «shift left» — сдвиг проверок безопасности как можно раньше в процессе разработки.

### 2. Проблема традиционного подхода к безопасности

При традиционном подходе проверка безопасности часто происходит в самом конце цикла разработки, непосредственно перед релизом — обнаруженные на этом этапе уязвимости требуют дорогостоящих и срочных исправлений, задерживающих выпуск продукта. Чем раньше обнаружена проблема безопасности, тем дешевле и быстрее её исправить.

### 3. Автоматизированное сканирование безопасности в CI/CD

DevSecOps встраивает автоматические проверки безопасности прямо в CI/CD пайплайн (Модуль 5):

- **SAST (Static Application Security Testing)** — статический анализ исходного кода на предмет уязвимостей без его запуска.
- **DAST (Dynamic Application Security Testing)** — тестирование уже запущенного приложения на уязвимости, имитируя действия атакующего.
- **Сканирование зависимостей (dependency scanning)** — проверка используемых сторонних библиотек на известные уязвимости (например, через `pip audit` для Python или Dependabot на GitHub).
- **Сканирование Docker-образов** — проверка образов контейнеров на известные уязвимости в базовых слоях перед их использованием в продакшене.

### 4. Пример: сканирование зависимостей в GitHub Actions

```yaml
- name: Проверка уязвимостей зависимостей
  run: |
    pip install pip-audit
    pip-audit
```

### 5. Управление секретами (повторение и расширение Урока 6.5)

DevSecOps уделяет особое внимание тому, чтобы пароли, ключи API и токены никогда не попадали в код репозитория — для этого используются инструменты сканирования репозитория на утечку секретов (например, GitHub Secret Scanning), автоматически предупреждающие при случайном коммите чувствительных данных.

### 6. Культурный аспект DevSecOps

Так же, как и DevOps в целом, DevSecOps — это не только набор инструментов, но и изменение культуры: ответственность за безопасность распределяется между всей командой разработки, а не считается исключительной задачей отдельного, изолированного отдела безопасности.

### Резюме урока

- DevSecOps встраивает проверки безопасности в процесс разработки максимально рано («shift left»), а не оставляет их на финальный этап перед релизом.
- SAST, DAST, сканирование зависимостей и Docker-образов — типичные автоматизированные проверки безопасности, встраиваемые в CI/CD пайплайн.
- Как и DevOps, DevSecOps — в первую очередь культурный сдвиг: ответственность за безопасность становится общей задачей всей команды.

## Урок 9.5. Практика: настройка простого дашборда мониторинга
**Время: 60 минут**

### Цель урока
Закрепить материал модуля, развернув базовую связку Prometheus + Grafana для мониторинга простого приложения через docker-compose.

### 1. Постановка задачи

Развернём Prometheus и Grafana через docker-compose (Урок 4.5) и настроим сбор базовых системных метрик с помощью вспомогательного экспортёра, а затем создадим простой дашборд для их визуализации.

### 2. Файл docker-compose.yml

```yaml
version: "3.9"

services:
  prometheus:
    image: prom/prometheus
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
    ports:
      - "9090:9090"

  node-exporter:
    image: prom/node-exporter
    ports:
      - "9100:9100"

  grafana:
    image: grafana/grafana
    ports:
      - "3000:3000"
    depends_on:
      - prometheus
```

`node-exporter` — стандартный вспомогательный экспортёр, предоставляющий базовые системные метрики (CPU, память, диск) в формате, понятном Prometheus.

### 3. Файл конфигурации prometheus.yml

```yaml
global:
  scrape_interval: 15s

scrape_configs:
  - job_name: "node-exporter"
    static_configs:
      - targets: ["node-exporter:9100"]
```

Этот файл указывает Prometheus, что нужно опрашивать метрики у `node-exporter` каждые 15 секунд.

### 4. Запуск и проверка

```bash
docker compose up -d
```

- Prometheus доступен на `http://localhost:9090` — можно проверить статус источников метрик во вкладке «Status → Targets».
- Grafana доступна на `http://localhost:3000` (данные для входа по умолчанию: admin/admin).

### 5. Подключение Prometheus как источника данных в Grafana

1. В интерфейсе Grafana перейти в «Connections → Data sources».
2. Добавить новый источник типа «Prometheus».
3. Указать URL: `http://prometheus:9090` (имя сервиса из docker-compose, а не `localhost`, так как Grafana обращается к Prometheus изнутри общей Docker-сети).

### 6. Создание простого дашборда

1. Создать новый дашборд («Dashboards → New Dashboard»).
2. Добавить панель (Panel) с запросом PromQL, например:

```promql
rate(node_cpu_seconds_total{mode="idle"}[5m])
```

3. Настроить визуализацию (тип графика, подписи осей) и сохранить дашборд.

### 7. Что демонстрирует эта практика

- Полный цикл настройки мониторинга: сбор метрик (node-exporter) → хранение (Prometheus) → визуализация (Grafana).
- Применение знаний docker-compose (Модуль 4) для быстрого развёртывания всей системы мониторинга.
- Базовое понимание PromQL для построения собственных запросов к метрикам.

### Резюме урока

- Связка Prometheus + Grafana, развёрнутая через docker-compose, позволяет быстро получить рабочую систему мониторинга для практики.
- Экспортёры (например, node-exporter) собирают метрики конкретной системы и предоставляют их в формате, понятном Prometheus.
- Настройка дашборда в Grafana включает подключение источника данных и создание панелей с запросами на PromQL.
