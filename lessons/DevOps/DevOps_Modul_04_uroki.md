# Модуль 4. Контейнеризация: Docker

## Урок 4.1. Проблема «у меня работает» и идея контейнеризации
**Время: 35 минут**

### Цель урока
Понять проблему, которую решает контейнеризация, и получить общее представление о принципах работы Docker.

### 1. Классическая проблема разработки

Ситуация, знакомая почти каждому разработчику: программа отлично работает на компьютере разработчика, но «падает» при развёртывании на сервере. Причины могут быть разными: другая версия языка программирования, отсутствующие зависимости, отличающиеся настройки операционной системы. Эта проблема получила ироничное название «у меня работает» (it works on my machine).

### 2. Как решали эту проблему раньше

- **Подробная документация по настройке окружения** — ненадёжно, документация быстро устаревает, легко допустить ошибку при ручной настройке.
- **Виртуальные машины** — полноценная эмуляция целой операционной системы для каждого приложения; надёжно, но очень «тяжело» с точки зрения потребления ресурсов и времени запуска.

### 3. Идея контейнеризации

Контейнер — это изолированная среда выполнения, которая содержит приложение вместе со всеми его зависимостями (библиотеки, настройки, переменные окружения), но при этом использует общее ядро операционной системы хоста, а не эмулирует полностью отдельную ОС, как виртуальная машина. Это делает контейнеры значительно легче и быстрее, чем виртуальные машины, сохраняя при этом изоляцию окружения.

### 4. Контейнеры vs виртуальные машины

| | Виртуальная машина | Контейнер |
|---|---|---|
| Что виртуализируется | Целая операционная система | Только приложение и его окружение |
| Размер | Гигабайты | Обычно десятки–сотни мегабайт |
| Время запуска | Минуты | Секунды |
| Изоляция | Полная (отдельное ядро ОС) | На уровне процессов (общее ядро ОС хоста) |

### 5. Что такое Docker

Docker — самая популярная платформа для создания, запуска и управления контейнерами. Docker позволяет описать окружение приложения один раз (в специальном файле — Dockerfile, подробнее в Уроке 4.4) и затем запускать абсолютно идентичное окружение на любой машине, где установлен Docker — будь то ноутбук разработчика, тестовый сервер или продакшен в облаке.

### 6. Ключевые преимущества контейнеризации для DevOps

- **Воспроизводимость.** Приложение ведёт себя одинаково в любом окружении.
- **Изоляция.** Разные приложения на одном сервере не мешают друг другу и их зависимостям.
- **Быстрое масштабирование.** Запуск дополнительных копий контейнера занимает секунды.
- **Упрощение CI/CD.** Приложение и его тестовое окружение можно упаковать в один контейнер, используемый на всех этапах пайплайна.

### Резюме урока

- Проблема «у меня работает» возникает из-за расхождений в окружении между машиной разработчика и сервером.
- Контейнеризация решает эту проблему, упаковывая приложение вместе со всеми зависимостями в переносимую изолированную среду.
- Контейнеры значительно легче и быстрее виртуальных машин, так как используют общее ядро операционной системы хоста.

## Урок 4.2. Установка Docker, основные команды (run, ps, stop)
**Время: 45 минут**

### Цель урока
Установить Docker и освоить базовые команды для запуска и управления контейнерами.

### 1. Установка Docker

**Windows/macOS:** скачать и установить Docker Desktop с официального сайта docker.com.

**Linux (Ubuntu):**
```bash
sudo apt update
sudo apt install docker.io
sudo systemctl start docker
sudo systemctl enable docker
```

Проверка установки:
```bash
docker --version
docker run hello-world
```

Команда `docker run hello-world` скачивает и запускает тестовый образ, подтверждающий, что Docker установлен и работает корректно.

### 2. Запуск первого реального контейнера

```bash
docker run nginx
```

Эта команда скачает образ веб-сервера nginx (если он ещё не скачан) и запустит контейнер на его основе.

### 3. Запуск с публикацией портов

Чтобы получить доступ к приложению внутри контейнера из браузера, нужно «пробросить» порт:

```bash
docker run -p 8080:80 nginx
```

Эта команда связывает порт 80 внутри контейнера (стандартный порт nginx) с портом 8080 на компьютере хоста — теперь приложение доступно по адресу `http://localhost:8080`.

### 4. Запуск в фоновом режиме

```bash
docker run -d -p 8080:80 nginx
```

Флаг `-d` (detached) запускает контейнер в фоновом режиме, освобождая терминал для дальнейшей работы.

### 5. Просмотр запущенных контейнеров: docker ps

```bash
docker ps            # список запущенных контейнеров
docker ps -a          # список всех контейнеров, включая остановленные
```

### 6. Остановка и удаление контейнеров

```bash
docker stop <container_id>       # остановить контейнер
docker start <container_id>       # запустить остановленный контейнер снова
docker rm <container_id>            # удалить контейнер (должен быть остановлен)
docker rm -f <container_id>         # принудительно остановить и удалить
```

Идентификатор контейнера (`container_id`) можно скопировать из вывода `docker ps`, при этом достаточно первых нескольких символов.

### 7. Именование контейнеров

Для удобства можно давать контейнерам понятные имена вместо использования автоматически сгенерированных идентификаторов:

```bash
docker run -d -p 8080:80 --name my_nginx nginx
docker stop my_nginx
```

### Резюме урока

- `docker run` создаёт и запускает контейнер на основе указанного образа, `-p` пробрасывает порты, `-d` запускает в фоновом режиме.
- `docker ps` показывает запущенные контейнеры, `docker ps -a` — все контейнеры, включая остановленные.
- `docker stop`/`docker start`/`docker rm` управляют жизненным циклом контейнера.

## Урок 4.3. Образы и контейнеры: разница, работа с Docker Hub
**Время: 40 минут**

### Цель урока
Понять разницу между образом и контейнером и научиться работать с публичным реестром образов Docker Hub.

### 1. Разница между образом и контейнером

- **Образ (image)** — это неизменяемый «шаблон» или «чертёж», содержащий приложение и все его зависимости.
- **Контейнер (container)** — это запущенный экземпляр образа, «живой» процесс, который можно запускать, останавливать и удалять.

Аналогия из ООП (Модуль 7 курса Python): образ похож на класс, а контейнер — на конкретный объект, созданный на его основе. От одного образа можно запустить сколько угодно контейнеров одновременно.

### 2. Просмотр загруженных образов

```bash
docker images
```

### 3. Загрузка образа: docker pull

```bash
docker pull python:3.12
docker pull node:20
docker pull postgres:16
```

### 4. Что такое Docker Hub

Docker Hub (hub.docker.com) — крупнейший публичный реестр Docker-образов, где хранятся официальные образы популярного ПО (Python, Node.js, PostgreSQL, nginx и т.д.), а также образы, опубликованные сообществом.

### 5. Теги образов (версии)

Образы на Docker Hub обычно имеют несколько тегов, соответствующих разным версиям:

```bash
docker pull python:3.12       # конкретная версия
docker pull python:latest      # последняя доступная версия
docker pull python:3.12-slim   # облегчённая версия образа
```

Использование конкретной версии тега (например, `python:3.12`) вместо `latest` — хорошая практика для продакшен-окружений, так как гарантирует предсказуемость поведения приложения при обновлениях.

### 6. Удаление образов

```bash
docker rmi python:3.12          # удалить конкретный образ
docker image prune               # удалить все неиспользуемые образы
```

### 7. Практический пример: запуск базы данных PostgreSQL

```bash
docker run -d \
  --name my_postgres \
  -e POSTGRES_PASSWORD=secret \
  -p 5432:5432 \
  postgres:16
```

Флаг `-e` устанавливает переменную окружения внутри контейнера — в данном случае пароль базы данных, который сама программа `postgres` ожидает найти в этой переменной при первом запуске.

### Резюме урока

- Образ — неизменяемый шаблон приложения, контейнер — запущенный на его основе экземпляр.
- Docker Hub — крупнейший публичный реестр образов, откуда образы скачиваются командой `docker pull`.
- Использование конкретных версий тегов образов (а не `latest`) — рекомендуемая практика для предсказуемости продакшен-окружений.

## Урок 4.4. Написание Dockerfile
**Время: 55 минут**

### Цель урока
Научиться описывать собственный Docker-образ с помощью Dockerfile.

### 1. Что такое Dockerfile

Dockerfile — это текстовый файл с набором инструкций, описывающих, как собрать Docker-образ: с какого базового образа начать, какие файлы скопировать, какие команды выполнить, что запускать при старте контейнера.

### 2. Базовая структура Dockerfile

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install -r requirements.txt --no-cache-dir

COPY . .

CMD ["python", "app.py"]
```

### 3. Разбор основных инструкций

| Инструкция | Назначение |
|---|---|
| `FROM` | указывает базовый образ, на основе которого строится новый |
| `WORKDIR` | задаёт рабочую директорию внутри контейнера |
| `COPY` | копирует файлы с локальной машины внутрь образа |
| `RUN` | выполняет команду во время сборки образа (например, установку зависимостей) |
| `CMD` | команда, выполняемая при запуске контейнера (может быть только одна) |
| `EXPOSE` | документирует, какой порт использует приложение внутри контейнера |
| `ENV` | задаёт переменную окружения внутри образа |

### 4. Сборка образа: docker build

```bash
docker build -t my_app:1.0 .
```

Флаг `-t` задаёт имя и тег образа, точка `.` в конце указывает контекст сборки (обычно текущая папка, где находится Dockerfile).

### 5. Запуск собственного образа

```bash
docker run -d -p 5000:5000 --name my_app_container my_app:1.0
```

### 6. Оптимизация Dockerfile: порядок инструкций и кэширование слоёв

Docker кэширует каждый шаг сборки как отдельный «слой». Если файл не изменился, соответствующий слой переиспользуется из кэша, что значительно ускоряет повторную сборку. Именно поэтому в примере из пункта 2 сначала копируется `requirements.txt` и устанавливаются зависимости, и только потом копируется весь остальной код — так изменение кода приложения не требует переустановки зависимостей при каждой сборке.

### 7. Пример Dockerfile для Node.js-приложения

```dockerfile
FROM node:20-slim

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .

EXPOSE 3000
CMD ["node", "server.js"]
```

### Резюме урока

- Dockerfile описывает пошаговую инструкцию сборки Docker-образа: базовый образ, копирование файлов, установка зависимостей, команда запуска.
- `docker build -t имя:тег .` собирает образ на основе Dockerfile из текущей папки.
- Правильный порядок инструкций (сначала зависимости, потом код) ускоряет повторную сборку за счёт кэширования слоёв Docker.

## Урок 4.5. docker-compose: запуск многоконтейнерных приложений
**Время: 50 минут**

### Цель урока
Освоить docker-compose — инструмент для описания и одновременного запуска нескольких связанных контейнеров.

### 1. Проблема, которую решает docker-compose

Реальные приложения редко состоят из одного контейнера — типичный веб-сервис включает как минимум приложение и базу данных, часто также кэш (Redis) и веб-сервер (nginx). Запускать и настраивать каждый контейнер отдельными длинными командами `docker run` неудобно и подвержено ошибкам.

`docker-compose` позволяет описать все контейнеры проекта и их взаимодействие в одном YAML-файле, а затем запускать (и останавливать) их все одной командой.

### 2. Базовая структура docker-compose.yml

```yaml
version: "3.9"

services:
  web:
    build: .
    ports:
      - "5000:5000"
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - db_data:/var/lib/postgresql/data

volumes:
  db_data:
```

### 3. Разбор структуры файла

- **services** — список контейнеров (сервисов), составляющих приложение.
- **build** — указывает, что образ для этого сервиса нужно собрать из Dockerfile в текущей папке (в отличие от `image`, который скачивает готовый образ).
- **depends_on** — указывает порядок запуска (сервис `web` запустится после `db`, хотя это не гарантирует полной готовности базы данных к моменту старта приложения).
- **volumes** — механизм сохранения данных вне жизненного цикла контейнера (подробнее ниже).

### 4. Основные команды docker-compose

```bash
docker compose up              # запустить все сервисы (в современных версиях Docker без дефиса)
docker compose up -d            # запустить в фоновом режиме
docker compose down             # остановить и удалить все контейнеры проекта
docker compose ps                # список запущенных сервисов проекта
docker compose logs web          # просмотр логов конкретного сервиса
```

### 5. Тома (volumes) — сохранение данных

По умолчанию все данные внутри контейнера теряются при его удалении. Том (volume) — это способ сохранить данные вне жизненного цикла контейнера, например, чтобы данные базы данных не пропадали при пересоздании контейнера:

```yaml
volumes:
  - db_data:/var/lib/postgresql/data
```

### 6. Сети (networks) в docker-compose

По умолчанию `docker-compose` создаёт общую сеть для всех сервисов, описанных в одном файле, — они могут обращаться друг к другу по имени сервиса (например, приложение `web` может подключаться к базе данных по адресу `db:5432`, используя имя сервиса вместо IP-адреса).

### 7. Практический пример: полное описание веб-приложения с базой данных

```yaml
version: "3.9"

services:
  web:
    build: .
    ports:
      - "5000:5000"
    environment:
      DATABASE_URL: postgresql://user:secret@db:5432/mydb
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: mydb
    volumes:
      - db_data:/var/lib/postgresql/data

volumes:
  db_data:
```

### Резюме урока

- `docker-compose` описывает несколько связанных контейнеров в едином YAML-файле и позволяет управлять ими одной командой.
- Сервисы в одном compose-файле автоматически объединены общей сетью и могут обращаться друг к другу по имени сервиса.
- Тома (volumes) сохраняют данные (например, содержимое базы данных) вне жизненного цикла отдельного контейнера.

## Урок 4.6. Практика: контейнеризация простого веб-приложения
**Время: 50 минут**

### Цель урока
Закрепить материал модуля, самостоятельно контейнеризировав небольшое веб-приложение с базой данных.

### 1. Постановка задачи

Возьмём простое Flask-приложение (Python), подключающееся к базе данных PostgreSQL, и полностью контейнеризируем его: напишем Dockerfile для приложения и docker-compose.yml для запуска всей системы одной командой.

### 2. Структура проекта

```
flask_app/
├── app.py
├── requirements.txt
├── Dockerfile
└── docker-compose.yml
```

### 3. Пример приложения (app.py)

```python
from flask import Flask
import os

app = Flask(__name__)

@app.route("/")
def home():
    return "Приложение успешно запущено в Docker-контейнере!"

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)
```

### 4. Файл зависимостей (requirements.txt)

```
flask==3.0.0
psycopg2-binary==2.9.9
```

### 5. Dockerfile для приложения

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install -r requirements.txt --no-cache-dir

COPY . .

EXPOSE 5000
CMD ["python", "app.py"]
```

### 6. docker-compose.yml

```yaml
version: "3.9"

services:
  web:
    build: .
    ports:
      - "5000:5000"
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - db_data:/var/lib/postgresql/data

volumes:
  db_data:
```

### 7. Запуск и проверка

```bash
docker compose up -d
docker compose ps
```

После запуска приложение должно быть доступно по адресу `http://localhost:5000`.

### 8. Проверка логов при возникновении проблем

```bash
docker compose logs web
docker compose logs db
```

### Резюме урока

- Полная контейнеризация приложения включает Dockerfile (описание образа приложения) и docker-compose.yml (описание всей системы сервисов).
- `docker compose up -d` запускает всю систему приложение + база данных одной командой.
- Просмотр логов через `docker compose logs` — основной способ диагностики проблем при запуске многоконтейнерных приложений, эта тема будет углублена в Модуле 9.
