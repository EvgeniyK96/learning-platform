# Модуль 5. CI/CD

---

## Урок 5.1. Что такое CI/CD и зачем это нужно
**Время: 35 минут**

### Цель урока
Углубить понимание концепций непрерывной интеграции и непрерывной доставки, начатое в Уроке 1.3, и разобрать их на практическом уровне.

### 1. Повторение: CI и CD

- **CI (Continuous Integration)** — практика частого объединения изменений кода в общую ветку с автоматическим запуском сборки и тестов при каждом таком объединении.
- **CD (Continuous Delivery/Deployment)** — практика автоматической подготовки (Delivery) или полностью автоматического выпуска (Deployment) проверенного кода в продакшен.

### 2. Что такое пайплайн (pipeline)

Пайплайн — это последовательность автоматизированных шагов, через которые проходит код от коммита до развёртывания. Типичный пайплайн включает этапы: сборка → тестирование → анализ качества кода → упаковка (например, в Docker-образ) → развёртывание.

### 3. Жизнь без CI/CD

Без автоматизации каждый релиз требует ручной сборки, ручного тестирования и ручного развёртывания — это медленно, требует много времени разработчиков и подвержено человеческим ошибкам. Проблемы совместимости кода часто обнаруживаются поздно — когда изменения от разных разработчиков наконец объединяются вручную перед релизом.

### 4. Преимущества CI/CD

- **Быстрое обнаружение ошибок.** Проблемы находятся сразу после коммита, а не через недели при подготовке релиза.
- **Меньший риск при релизе.** Частые небольшие изменения проще откатить и легче отследить причину проблемы, чем один крупный релиз раз в месяц.
- **Экономия времени команды.** Рутинные операции сборки, тестирования и развёртывания выполняются автоматически.
- **Уверенность в качестве кода.** Автоматические тесты запускаются при каждом изменении, снижая риск случайно сломать существующий функционал.

### 5. Триггеры запуска пайплайна

Пайплайн CI/CD обычно запускается автоматически при определённых событиях в репозитории:
- При каждом push в определённую ветку.
- При создании или обновлении Pull Request.
- По расписанию (например, ежедневная сборка).
- Вручную, по требованию.

### 6. Основные инструменты CI/CD

| Инструмент | Особенности |
|---|---|
| GitHub Actions | Встроен в GitHub, конфигурация через YAML-файлы в репозитории |
| GitLab CI/CD | Встроен в GitLab, аналогичный подход |
| Jenkins | Независимый инструмент с открытым исходным кодом, требует отдельной установки и настройки сервера |
| CircleCI | Облачный сервис CI/CD |

В следующих уроках модуля мы подробно разберём GitHub Actions как наиболее доступный инструмент для старта.

### Резюме урока

- CI/CD пайплайн — последовательность автоматизированных шагов от коммита кода до его развёртывания.
- CI/CD значительно ускоряет обнаружение ошибок и снижает риски, связанные с релизами.
- Пайплайны обычно запускаются автоматически по событиям в репозитории: push, Pull Request, по расписанию.

## Урок 5.2. Обзор инструментов: GitHub Actions, GitLab CI, Jenkins
**Время: 40 минут**

### Цель урока
Сравнить основные инструменты CI/CD и понять, в каких ситуациях каждый из них предпочтителен.

### 1. GitHub Actions

Встроенный в GitHub сервис CI/CD. Конфигурация описывается в YAML-файлах внутри папки `.github/workflows/` репозитория. Не требует отдельной установки — работает «из коробки» для любого репозитория на GitHub.

**Плюсы:** тесная интеграция с GitHub, большой каталог готовых переиспользуемых действий (Actions) от сообщества, бесплатный лимит для публичных и (в разумных пределах) приватных репозиториев.

**Минусы:** привязан к экосистеме GitHub.

### 2. GitLab CI/CD

Встроенный в GitLab сервис CI/CD, аналогичный по концепции GitHub Actions. Конфигурация описывается в файле `.gitlab-ci.yml` в корне репозитория.

**Плюсы:** тесная интеграция с GitLab, гибкая система runner'ов (в том числе можно разворачивать собственные), популярен в корпоративной среде с self-hosted GitLab.

**Минусы:** привязан к экосистеме GitLab.

### 3. Jenkins

Независимый, самый «взрослый» инструмент CI/CD с открытым исходным кодом, существующий с 2011 года. В отличие от GitHub Actions и GitLab CI, требует самостоятельной установки и администрирования собственного сервера Jenkins.

**Плюсы:** максимальная гибкость и настраиваемость, огромная экосистема плагинов, не привязан к конкретной платформе хранения кода.

**Минусы:** требует значительно больше усилий на установку, настройку и поддержку по сравнению с облачными решениями.

### 4. Сравнительная таблица

| Критерий | GitHub Actions | GitLab CI | Jenkins |
|---|---|---|---|
| Установка | Не требуется | Не требуется (для GitLab.com) | Требуется отдельный сервер |
| Конфигурация | YAML в репозитории | YAML в репозитории | Groovy-скрипты (Jenkinsfile) или UI |
| Гибкость | Средняя-высокая | Средняя-высокая | Максимальная |
| Порог входа | Низкий | Низкий | Средний-высокий |

### 5. Какой инструмент выбрать для старта

Для большинства новых проектов, особенно небольших и учебных, **GitHub Actions** — оптимальный выбор для начала: не требует установки, тесно интегрирован с местом хранения кода, имеет понятную документацию и большое сообщество. Именно его мы будем использовать в практических уроках этого модуля.

### 6. Общие концепции, применимые ко всем инструментам

Несмотря на различия в синтаксисе, все CI/CD инструменты оперируют схожими понятиями: **workflow/pipeline** (весь процесс), **job/stage** (отдельный этап, например «тестирование»), **step/task** (конкретное действие внутри этапа), **runner/agent** (машина, на которой выполняется пайплайн).

### Резюме урока

- GitHub Actions и GitLab CI — встроенные, не требующие отдельной установки решения, привязанные к соответствующей платформе.
- Jenkins — независимый, максимально гибкий, но требующий самостоятельной установки и администрирования инструмент.
- Для старта и учебных целей GitHub Actions — самый доступный и практичный выбор.

## Урок 5.3. Написание простого CI-пайплайна (GitHub Actions)
**Время: 55 минут**

### Цель урока
Написать и запустить первый рабочий CI-пайплайн с помощью GitHub Actions.

### 1. Структура файлов GitHub Actions

Конфигурация workflow (пайплайна) размещается в YAML-файлах внутри папки `.github/workflows/` в корне репозитория:

```
my_project/
├── .github/
│   └── workflows/
│       └── ci.yml
├── app.py
└── requirements.txt
```

### 2. Базовая структура workflow-файла

```yaml
name: CI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    
    steps:
      - name: Получение кода
        uses: actions/checkout@v4
      
      - name: Вывод приветствия
        run: echo "Пайплайн запущен успешно!"
```

### 3. Разбор ключевых секций

- **name** — название workflow, отображаемое в интерфейсе GitHub.
- **on** — события, при которых запускается пайплайн (в примере — push и Pull Request в ветку main).
- **jobs** — список задач (job), каждая из которых выполняется на отдельной виртуальной машине.
- **runs-on** — тип виртуальной машины, на которой будет выполняться задача (обычно `ubuntu-latest`).
- **steps** — последовательность шагов внутри задачи.
- **uses** — использование готового переиспользуемого действия из каталога GitHub Actions (например, `actions/checkout` для получения кода репозитория).
- **run** — выполнение произвольной команды терминала.

### 4. Практический пример: пайплайн для Python-проекта

```yaml
name: Python CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    
    steps:
      - name: Получение кода
        uses: actions/checkout@v4
      
      - name: Установка Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      
      - name: Установка зависимостей
        run: pip install -r requirements.txt
      
      - name: Проверка синтаксиса
        run: python -m py_compile app.py
```

### 5. Запуск и просмотр результатов

После добавления файла в репозиторий и выполнения `git push`, workflow автоматически запустится. Результат выполнения (успех/провал каждого шага, полные логи) можно увидеть во вкладке «Actions» репозитория на GitHub.

### 6. Запуск нескольких задач параллельно

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Тестирование"

  lint:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Проверка стиля кода"
```

Задачи `test` и `lint` в этом примере не зависят друг от друга и будут выполняться параллельно, ускоряя общее время работы пайплайна.

### Резюме урока

- Конфигурация GitHub Actions размещается в YAML-файлах внутри `.github/workflows/` в корне репозитория.
- Пайплайн состоит из задач (jobs), каждая из которых включает последовательность шагов (steps).
- Результаты выполнения пайплайна видны во вкладке «Actions» репозитория на GitHub.

## Урок 5.4. Автоматическое тестирование в пайплайне
**Время: 45 минут**

### Цель урока
Научиться подключать автоматические тесты к CI-пайплайну, чтобы каждое изменение кода автоматически проверялось.

### 1. Зачем тесты нужны именно в пайплайне

Наличие тестов в проекте (Урок 9.3 курса Python) само по себе полезно, но их истинная сила раскрывается, когда они запускаются автоматически при каждом изменении — это гарантирует, что ни один коммит не «сломает» существующий функционал незамеченным.

### 2. Пример: подключение unittest к пайплайну

Допустим, в проекте уже есть тесты, написанные через `unittest` (см. Урок 9.3 курса Python):

```yaml
name: Python CI with Tests

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v4
      
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      
      - name: Установка зависимостей
        run: pip install -r requirements.txt
      
      - name: Запуск тестов
        run: python -m unittest discover tests/
```

### 3. Обязательность прохождения тестов для слияния Pull Request

На GitHub можно настроить правило защиты ветки (Branch Protection Rule), запрещающее слияние Pull Request, пока все обязательные проверки (включая CI-пайплайн) не завершились успешно. Это гарантирует, что в основную ветку не попадёт код, ломающий тесты.

### 4. Матрица тестирования (testing matrix)

Иногда полезно проверить код на нескольких версиях языка или разных операционных системах одновременно:

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ["3.10", "3.11", "3.12"]
    
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - run: pip install -r requirements.txt
      - run: python -m unittest discover tests/
```

Такая матрица запустит тесты трижды — на каждой из трёх указанных версий Python — параллельно.

### 5. Проверка стиля кода (linting) в пайплайне

Помимо функциональных тестов, в пайплайн часто добавляют автоматическую проверку соответствия коду стандартам (PEP 8 для Python, ESLint для JavaScript):

```yaml
- name: Проверка стиля кода
  run: |
    pip install flake8
    flake8 . --max-line-length=100
```

### 6. Отчёт о покрытии кода тестами (code coverage)

Многие команды дополнительно отслеживают, какой процент кода фактически покрыт тестами, — это помогает находить недостаточно протестированные участки:

```yaml
- name: Тесты с покрытием
  run: |
    pip install coverage
    coverage run -m unittest discover tests/
    coverage report
```

### Резюме урока

- Подключение тестов к CI-пайплайну гарантирует, что каждое изменение кода автоматически проверяется перед слиянием.
- Правила защиты веток на GitHub позволяют запретить слияние Pull Request без успешного прохождения тестов.
- Матрица тестирования позволяет параллельно проверять код на нескольких версиях языка или окружениях.

## Урок 5.5. Автоматический деплой (CD) приложения
**Время: 55 минут**

### Цель урока
Расширить CI-пайплайн этапом автоматического развёртывания (CD) приложения.

### 1. От CI к CD

После того как код собран и протестирован (этапы CI), следующий шаг — автоматическая доставка этого кода в рабочее окружение (этап CD). Это может быть развёртывание на сервере, публикация Docker-образа в реестр или деплой на облачную платформу.

### 2. Секреты в GitHub Actions

Для развёртывания часто требуются конфиденциальные данные (пароли, токены доступа, SSH-ключи), которые нельзя хранить прямо в коде пайплайна. Для этого в настройках репозитория GitHub есть раздел «Secrets», куда можно добавить защищённые переменные, доступные пайплайну, но не отображаемые в логах:

```yaml
steps:
  - name: Деплой
    env:
      API_TOKEN: ${{ secrets.DEPLOY_API_TOKEN }}
    run: ./deploy.sh
```

### 3. Пример: сборка и публикация Docker-образа

```yaml
name: Build and Push Docker Image

on:
  push:
    branches: [main]

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Вход в Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}
      
      - name: Сборка и публикация образа
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: myusername/my_app:latest
```

### 4. Условное выполнение деплоя (только после успешных тестов)

Обычно деплой должен выполняться только после того, как этап тестирования завершился успешно — это реализуется через зависимость между задачами (jobs):

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Запуск тестов"

  deploy:
    needs: test          # deploy запустится только после успешного завершения test
    runs-on: ubuntu-latest
    steps:
      - run: echo "Развёртывание приложения"
```

### 5. Пример: развёртывание на сервер по SSH

```yaml
jobs:
  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - name: Деплой по SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SERVER_SSH_KEY }}
          script: |
            cd /var/www/my_app
            git pull origin main
            docker compose up -d --build
```

### 6. Continuous Delivery vs Continuous Deployment на практике

Если развёртывание в продакшен происходит автоматически при каждом успешном прохождении пайплайна из ветки `main` — это Continuous Deployment. Если же для финального шага деплоя требуется ручное подтверждение (например, через отдельную кнопку «Approve» в GitHub Environments) — это Continuous Delivery. Оба подхода валидны, выбор зависит от уровня доверия к автоматическим тестам и критичности приложения.

### Резюме урока

- Секреты в GitHub Actions (репозиторий → Settings → Secrets) позволяют безопасно использовать конфиденциальные данные в пайплайне.
- Ключевое слово `needs` устанавливает зависимость между задачами, гарантируя, что деплой запустится только после успешного тестирования.
- Continuous Deployment разворачивает изменения полностью автоматически, Continuous Delivery добавляет шаг ручного подтверждения перед финальным релизом.
