# Модуль 7. Оркестрация контейнеров: Kubernetes (основы)

---

## Урок 7.1. Зачем нужен Kubernetes, когда Docker уже есть
**Время: 35 минут**

### Цель урока
Понять, какую проблему решает Kubernetes, и почему одного Docker часто недостаточно для продакшен-инфраструктуры.

### 1. Ограничения Docker и docker-compose в масштабе

Docker и docker-compose отлично справляются с задачей запуска контейнеров на одной машине. Но что происходит, когда:
- Нужно распределить сотни контейнеров по десяткам серверов?
- Сервер, на котором работает контейнер, выходит из строя — кто перезапустит контейнер на другом сервере?
- Нагрузка на приложение резко выросла — как быстро запустить дополнительные копии контейнера?
- Нужно обновить приложение без остановки его работы для пользователей?

Docker и docker-compose не решают эти задачи «из коробки» — для этого нужен отдельный инструмент, оркестратор контейнеров.

### 2. Что такое Kubernetes

Kubernetes (часто сокращается как K8s) — система оркестрации контейнеров с открытым исходным кодом, изначально разработанная в Google на основе их внутреннего опыта управления контейнерами в огромном масштабе. Kubernetes автоматизирует развёртывание, масштабирование и управление контейнеризированными приложениями на кластере из множества серверов.

### 3. Ключевые задачи, которые решает Kubernetes

- **Автоматическое восстановление (self-healing).** Если контейнер или целый сервер выходит из строя, Kubernetes автоматически перезапускает приложение на исправном сервере.
- **Масштабирование.** Автоматическое увеличение или уменьшение числа запущенных копий приложения в зависимости от нагрузки.
- **Балансировка нагрузки.** Равномерное распределение входящего трафика между несколькими копиями приложения.
- **Обновление без простоя (rolling updates).** Постепенное обновление приложения новыми версиями контейнеров без прерывания его доступности для пользователей.
- **Управление конфигурацией и секретами.** Централизованное хранение настроек приложений, отделённое от самого кода.

### 4. Когда Kubernetes действительно нужен

Kubernetes добавляет значительную сложность, и для небольших проектов с одним-двумя серверами docker-compose часто более чем достаточен. Kubernetes становится оправданным выбором, когда:
- Приложение состоит из множества (десятков и более) независимых сервисов (микросервисная архитектура).
- Требуется высокая доступность и автоматическое восстановление после сбоев.
- Нагрузка сильно колеблется и требует динамического масштабирования.
- Инфраструктура распределена между множеством серверов.

### 5. Managed Kubernetes от облачных провайдеров

Самостоятельное развёртывание и обслуживание Kubernetes-кластера («с нуля») — сложная задача. Большинство компаний используют управляемые (managed) сервисы Kubernetes от облачных провайдеров — Amazon EKS, Google GKE, Azure AKS (Урок 6.2) — которые берут на себя администрирование самого «управляющего слоя» кластера.

### 6. Kubernetes и Docker вместе

Важно понимать: Kubernetes не заменяет Docker, а работает поверх него (или другой совместимой среды выполнения контейнеров) — Docker (или аналог) непосредственно запускает контейнеры на каждом сервере, а Kubernetes управляет тем, где, сколько и когда эти контейнеры должны быть запущены в масштабе всего кластера.

### Резюме урока

- Kubernetes — система оркестрации контейнеров, решающая задачи масштабирования, автоматического восстановления и управления множеством контейнеров на кластере серверов, которые не решает сам по себе Docker.
- Kubernetes оправдан для крупных, распределённых приложений с высокими требованиями к доступности; для небольших проектов часто достаточно docker-compose.
- Большинство компаний используют управляемые Kubernetes-сервисы облачных провайдеров, а не разворачивают кластер полностью самостоятельно.

## Урок 7.2. Архитектура Kubernetes: под, нода, кластер
**Время: 45 минут**

### Цель урока
Изучить базовые строительные блоки архитектуры Kubernetes.

### 1. Кластер (Cluster)

Кластер — это набор серверов (физических или виртуальных), объединённых для совместной работы под управлением Kubernetes. Кластер состоит из двух типов узлов: control plane (управляющий слой) и worker-узлы (рабочие узлы), на которых непосредственно выполняются приложения.

### 2. Control Plane (управляющий слой)

Управляющий слой — «мозг» кластера, отвечающий за принятие решений: где запускать контейнеры, как реагировать на сбои, как масштабировать приложения. При использовании managed-сервисов (Урок 7.1) облачный провайдер полностью берёт на себя администрирование control plane.

### 3. Нода (Node)

Нода — это отдельный сервер (физический или виртуальный) в составе кластера, на котором непосредственно запускаются контейнеры приложений. Кластер обычно состоит из нескольких worker-нод для обеспечения отказоустойчивости и распределения нагрузки.

### 4. Под (Pod) — минимальная единица развёртывания

Под — это наименьшая единица развёртывания в Kubernetes. Под содержит один или несколько тесно связанных контейнеров, которые всегда запускаются вместе на одной ноде и разделяют общую сеть и хранилище. В большинстве случаев под содержит ровно один контейнер приложения.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app-pod
spec:
  containers:
    - name: my-app
      image: my_app:1.0
      ports:
        - containerPort: 5000
```

### 5. Почему поды, а не контейнеры напрямую

Kubernetes оперирует подами, а не отдельными контейнерами, потому что некоторые приложения требуют нескольких тесно связанных процессов, работающих вместе (например, основное приложение и вспомогательный контейнер для сбора логов) — под позволяет объединить их как единое целое с общей сетью и жизненным циклом.

### 6. Эфемерность подов

Важная особенность: поды эфемерны — они могут быть удалены и пересозданы Kubernetes в любой момент (например, при перемещении на другую ноду или при обновлении приложения), при этом под получает новый уникальный идентификатор и, как правило, новый внутренний IP-адрес. Именно поэтому напрямую обращаться к конкретному поду — плохая практика; для этого существуют более стабильные абстракции, такие как Service (Урок 7.3).

### 7. Наглядная иерархия

```
Кластер
 ├── Control Plane (управляющий слой)
 └── Worker-ноды
      ├── Нода 1
      │    ├── Под А (контейнер приложения)
      │    └── Под Б (контейнер приложения)
      └── Нода 2
           └── Под В (контейнер приложения)
```

### Резюме урока

- Кластер Kubernetes состоит из control plane (принятие решений) и worker-нод (выполнение приложений).
- Под — минимальная единица развёртывания, содержащая один или несколько тесно связанных контейнеров.
- Поды эфемерны и могут пересоздаваться Kubernetes в любой момент, поэтому для стабильного доступа к приложению используются дополнительные абстракции более высокого уровня.

## Урок 7.3. Deployment и Service: базовые объекты
**Время: 50 минут**

### Цель урока
Изучить два ключевых объекта Kubernetes для управления приложениями: Deployment и Service.

### 1. Проблема прямого управления подами

Как отмечалось в предыдущем уроке, поды эфемерны — управлять ими напрямую в реальном приложении неудобно и ненадёжно. Для практического использования Kubernetes предоставляет объекты более высокого уровня, автоматизирующие управление подами.

### 2. Что такое Deployment

Deployment — объект Kubernetes, описывающий желаемое состояние приложения: какой образ использовать, сколько копий (реплик) пода должно быть запущено одновременно, как обновлять приложение. Kubernetes постоянно следит за тем, чтобы фактическое состояние соответствовало описанному в Deployment, автоматически пересоздавая поды при сбоях.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app
          image: my_app:1.0
          ports:
            - containerPort: 5000
```

Здесь `replicas: 3` означает, что Kubernetes должен постоянно поддерживать ровно 3 запущенные копии пода с этим приложением — если одна из них выйдет из строя, будет автоматически создана новая.

### 3. Что такое Service

Service — объект, предоставляющий стабильную точку доступа (постоянный внутренний IP-адрес и DNS-имя) к группе подов, даже если сами поды регулярно пересоздаются с новыми IP-адресами. Service автоматически распределяет входящий трафик между всеми доступными подами приложения (балансировка нагрузки).

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 5000
  type: ClusterIP
```

### 4. Типы Service

| Тип | Назначение |
|---|---|
| `ClusterIP` | доступ только изнутри кластера (по умолчанию) |
| `NodePort` | доступ через определённый порт на каждой ноде кластера |
| `LoadBalancer` | автоматическое создание внешнего балансировщика нагрузки (обычно у облачного провайдера) |

### 5. Как Deployment и Service связаны через labels (метки)

Service находит нужные ему поды не по имени, а через систему меток (labels) и селекторов (selectors) — в примерах выше и Deployment, и Service используют метку `app: my-app`, что позволяет Service автоматически «видеть» все поды, созданные этим Deployment, независимо от их конкретных, постоянно меняющихся имён.

### 6. Применение конфигурации через kubectl (подробнее в Уроке 7.4)

```bash
kubectl apply -f deployment.yaml
kubectl apply -f service.yaml
```

### Резюме урока

- Deployment описывает желаемое состояние приложения (образ, число реплик) и автоматически поддерживает его, пересоздавая поды при сбоях.
- Service предоставляет стабильную точку доступа к группе подов и распределяет между ними входящий трафик.
- Связь между Deployment и Service обеспечивается через систему меток (labels) и селекторов (selectors), а не через прямые ссылки на конкретные поды.

## Урок 7.4. Работа с kubectl
**Время: 45 минут**

### Цель урока
Освоить kubectl — основной инструмент командной строки для взаимодействия с Kubernetes-кластером.

### 1. Что такое kubectl

`kubectl` — консольная утилита для отправки команд Kubernetes-кластеру: создание, изменение, удаление и просмотр объектов (подов, Deployment, Service и т.д.).

### 2. Установка и настройка kubectl (обзорно)

Установка `kubectl` зависит от операционной системы (доступны инструкции на официальном сайте kubernetes.io). Для локальной практики без реального облачного кластера часто используется **Minikube** — инструмент, разворачивающий небольшой Kubernetes-кластер прямо на локальном компьютере:

```bash
minikube start
```

### 3. Просмотр объектов кластера

```bash
kubectl get pods                  # список подов
kubectl get deployments            # список Deployment
kubectl get services                # список Service
kubectl get nodes                   # список нод кластера
```

### 4. Подробная информация об объекте

```bash
kubectl describe pod my-app-pod-xyz
```

Команда `describe` показывает подробную информацию об объекте, включая события — крайне полезно для диагностики проблем (например, почему под не запускается).

### 5. Применение и удаление конфигурации

```bash
kubectl apply -f deployment.yaml     # создать/обновить объект согласно файлу
kubectl delete -f deployment.yaml     # удалить объект, описанный в файле
kubectl delete pod my-app-pod-xyz      # удалить конкретный под напрямую
```

### 6. Просмотр логов пода

```bash
kubectl logs my-app-pod-xyz
kubectl logs -f my-app-pod-xyz        # непрерывный просмотр (аналог tail -f)
```

### 7. Выполнение команды внутри пода

```bash
kubectl exec -it my-app-pod-xyz -- /bin/bash
```

Эта команда открывает интерактивную сессию терминала внутри работающего контейнера пода — полезно для отладки в реальном времени.

### 8. Проброс порта для локального доступа

```bash
kubectl port-forward service/my-app-service 8080:80
```

Позволяет временно обратиться к сервису внутри кластера через `localhost:8080` на локальной машине, без необходимости настройки полноценного внешнего доступа — удобно для тестирования.

### Резюме урока

- `kubectl` — основной инструмент управления Kubernetes-кластером из командной строки: `get`, `describe`, `apply`, `delete`, `logs`, `exec`.
- Minikube позволяет развернуть небольшой Kubernetes-кластер локально для практики без облачных затрат.
- `kubectl logs` и `kubectl describe` — основные инструменты диагностики проблем с подами.

## Урок 7.5. Масштабирование и обновление приложений
**Время: 45 минут**

### Цель урока
Научиться масштабировать приложения в Kubernetes и обновлять их без прерывания работы.

### 1. Ручное масштабирование

```bash
kubectl scale deployment my-app-deployment --replicas=5
```

Эта команда изменяет число запущенных копий (реплик) пода на лету, без необходимости редактировать и повторно применять YAML-файл.

### 2. Автоматическое масштабирование: Horizontal Pod Autoscaler (HPA)

HPA автоматически изменяет число реплик Deployment в зависимости от текущей нагрузки (обычно на основе использования CPU или памяти):

```bash
kubectl autoscale deployment my-app-deployment --min=2 --max=10 --cpu-percent=70
```

Эта команда настраивает автоматическое масштабирование от 2 до 10 реплик, ориентируясь на целевую загрузку CPU в 70%.

### 3. Обновление приложения: изменение образа

Самый частый сценарий обновления — выпуск новой версии контейнера приложения:

```bash
kubectl set image deployment/my-app-deployment my-app=my_app:2.0
```

### 4. Rolling Update — обновление без простоя

По умолчанию Kubernetes обновляет поды Deployment постепенно (rolling update): не все поды заменяются одновременно, а по очереди — сначала создаётся новый под с обновлённой версией, и только после того, как он успешно запустился, останавливается один из старых. Это обеспечивает непрерывную доступность приложения во время обновления.

### 5. Просмотр статуса обновления

```bash
kubectl rollout status deployment/my-app-deployment
```

### 6. Откат к предыдущей версии

Если новая версия оказалась неработоспособной, Kubernetes позволяет быстро откатиться к предыдущей версии Deployment:

```bash
kubectl rollout undo deployment/my-app-deployment
```

```bash
kubectl rollout history deployment/my-app-deployment   # просмотр истории версий
```

### 7. Проверка работоспособности: readiness и liveness probes (обзорно)

Для того чтобы Kubernetes корректно понимал, готов ли под принимать трафик и жив ли он, используются специальные проверки:

```yaml
containers:
  - name: my-app
    image: my_app:2.0
    livenessProbe:
      httpGet:
        path: /health
        port: 5000
      initialDelaySeconds: 5
      periodSeconds: 10
```

`livenessProbe` периодически проверяет, жив ли контейнер (и перезапускает его при неудаче), `readinessProbe` — готов ли он принимать трафик (и временно исключает под из балансировки, если нет).

### Резюме урока

- Масштабирование в Kubernetes может быть ручным (`kubectl scale`) или автоматическим на основе нагрузки (Horizontal Pod Autoscaler).
- Rolling update по умолчанию обновляет приложение постепенно, без простоя для пользователей.
- `kubectl rollout undo` позволяет быстро откатиться к предыдущей рабочей версии приложения в случае проблем.

## Урок 7.6. Практика: развёртывание приложения в Kubernetes (Minikube)
**Время: 60 минут**

### Цель урока
Закрепить материал модуля, развернув полноценное приложение в локальном Kubernetes-кластере через Minikube.

### 1. Подготовка окружения

```bash
minikube start
kubectl get nodes     # проверка, что кластер запущен
```

### 2. Использование образа приложения из Модуля 4

Для практики возьмём тот же Flask-образ, что был собран в Уроке 4.6 (`my_app:1.0`). При работе с Minikube образ нужно либо загрузить в его собственное окружение Docker, либо опубликовать в реестре, доступном кластеру:

```bash
eval $(minikube docker-env)     # переключаемся на Docker-окружение Minikube
docker build -t my_app:1.0 .      # собираем образ уже внутри окружения Minikube
```

### 3. Файл Deployment (deployment.yaml)

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: flask-app-deployment
spec:
  replicas: 2
  selector:
    matchLabels:
      app: flask-app
  template:
    metadata:
      labels:
        app: flask-app
    spec:
      containers:
        - name: flask-app
          image: my_app:1.0
          imagePullPolicy: Never   # использовать локальный образ, не искать в реестре
          ports:
            - containerPort: 5000
```

### 4. Файл Service (service.yaml)

```yaml
apiVersion: v1
kind: Service
metadata:
  name: flask-app-service
spec:
  selector:
    app: flask-app
  ports:
    - port: 80
      targetPort: 5000
  type: NodePort
```

### 5. Применение конфигурации и проверка

```bash
kubectl apply -f deployment.yaml
kubectl apply -f service.yaml

kubectl get pods
kubectl get services
```

### 6. Доступ к приложению

```bash
minikube service flask-app-service
```

Эта команда Minikube автоматически откроет приложение в браузере через временно проброшенный порт.

### 7. Эксперименты для закрепления материала

- Масштабируй приложение до 4 реплик через `kubectl scale` и убедись, что появились новые поды.
- Удали один из подов вручную (`kubectl delete pod <имя>`) и понаблюдай, как Kubernetes автоматически создаёт новый взамен.
- Просмотри логи одного из подов через `kubectl logs`.

### Резюме урока

- Minikube — удобный способ практиковаться с Kubernetes локально, без затрат на облачную инфраструктуру.
- Полное развёртывание приложения включает Deployment (управление подами) и Service (стабильная точка доступа и балансировка).
- Эксперименты с ручным удалением подов наглядно демонстрируют ключевое преимущество Kubernetes — автоматическое самовосстановление (self-healing).
