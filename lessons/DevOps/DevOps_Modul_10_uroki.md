# Модуль 10. Итоговый проект

---

## Урок 10.1. Планирование: сборка полного CI/CD пайплайна для приложения
**Время: 45 минут**

### Цель урока
Спланировать итоговый проект курса — полный CI/CD пайплайн для реального приложения, объединяющий все изученные темы.

### 1. Задача итогового проекта

Итоговый проект курса — собрать полный цикл DevOps-практик для простого приложения: от кода в Git до автоматического развёртывания, с контейнеризацией, CI/CD пайплайном и базовым мониторингом.

### 2. Выбор приложения для проекта

Для итогового проекта не нужно писать сложное приложение — подойдёт простое веб-приложение (например, на Flask/Python или Express/Node.js) с одним-двумя эндпоинтами, включая проверочный `/health`-эндпоинт для будущих readiness/liveness проверок (Урок 7.5).

### 3. Структура технического задания

1. **Приложение.** Какое простое приложение будет использовано (можно взять готовый пример из Модуля 4 или написать новое).
2. **Инфраструктура.** Где будет развёрнуто приложение (облачный сервер, локальный Kubernetes через Minikube).
3. **CI/CD пайплайн.** Какие этапы должен включать: сборка, тесты, публикация Docker-образа, деплой.
4. **Мониторинг.** Как будет отслеживаться состояние приложения после развёртывания.

### 4. Пример технического задания

**Проект:** CI/CD пайплайн для простого Flask-приложения

**Компоненты:**
- Flask-приложение с эндпоинтами `/` и `/health`, контейнеризированное через Dockerfile (Модуль 4).
- Git-репозиторий на GitHub с настроенными ветками и Branch Protection Rule (Модуль 3).
- GitHub Actions пайплайн: тесты → сборка Docker-образа → публикация в Docker Hub → деплой (Модуль 5).
- Развёртывание на облачном сервере (Модуль 6) или в Minikube (Модуль 7).
- Базовый мониторинг через Prometheus + Grafana (Модуль 9).

### 5. Итоговая архитектурная схема

```
Разработчик → git push → GitHub
                              │
                              ▼
                    GitHub Actions (CI)
                    ├── Тесты
                    ├── Сборка Docker-образа
                    └── Публикация в Docker Hub
                              │
                              ▼
                    GitHub Actions (CD)
                    └── Деплой на сервер / в Kubernetes
                              │
                              ▼
                    Работающее приложение
                              │
                              ▼
                Prometheus + Grafana (мониторинг)
```

### 6. Оценка объёма задачи

Итоговый проект должен быть выполним за отведённое время модуля (около 5 часов) и явно демонстрировать минимум 5–6 тем курса. Не обязательно реализовывать абсолютно все возможные компоненты (например, Kubernetes и Terraform можно оставить опциональными, если позволяет время) — важнее продемонстрировать целостное понимание связи между этапами DevOps-цикла.

### Резюме урока

- Итоговый проект должен объединять реальные DevOps-практики в единый работающий цикл: код → CI → сборка образа → CD → мониторинг.
- Техническое задание чётко определяет приложение, инфраструктуру, этапы пайплайна и подход к мониторингу до начала практической реализации.
- Не обязательно реализовывать абсолютно все темы курса — важнее продемонстрировать понимание целостной картины DevOps-цикла.

## Урок 10.2. Контейнеризация приложения и настройка Docker Compose
**Время: 60 минут**

### Цель урока
Реализовать первый этап итогового проекта: подготовить приложение и контейнеризировать его.

### 1. Подготовка простого приложения

```python
# app.py
from flask import Flask, jsonify
import os

app = Flask(__name__)

@app.route("/")
def home():
    return jsonify({"message": "Приложение работает!", "version": os.getenv("APP_VERSION", "1.0")})

@app.route("/health")
def health():
    return jsonify({"status": "ok"}), 200

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)
```

### 2. Файл зависимостей

```
flask==3.0.0
```

### 3. Написание тестов (для последующего использования в CI)

```python
# test_app.py
import unittest
from app import app

class TestApp(unittest.TestCase):
    def setUp(self):
        self.client = app.test_client()

    def test_home(self):
        response = self.client.get("/")
        self.assertEqual(response.status_code, 200)

    def test_health(self):
        response = self.client.get("/health")
        self.assertEqual(response.status_code, 200)
        self.assertEqual(response.json["status"], "ok")

if __name__ == "__main__":
    unittest.main()
```

### 4. Dockerfile

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install -r requirements.txt --no-cache-dir

COPY . .

EXPOSE 5000
CMD ["python", "app.py"]
```

### 5. docker-compose.yml для локальной разработки

```yaml
version: "3.9"

services:
  web:
    build: .
    ports:
      - "5000:5000"
    environment:
      APP_VERSION: "1.0"
```

### 6. Проверка локальной работы

```bash
docker compose up -d
curl http://localhost:5000/
curl http://localhost:5000/health
```

### 7. Инициализация Git-репозитория

```bash
git init
git add .
git commit -m "Начальная версия приложения с контейнеризацией"
git remote add origin https://github.com/username/devops-final-project.git
git push -u origin main
```

### Резюме урока

- Первый этап итогового проекта — подготовка простого работающего приложения с базовыми тестами и его контейнеризация через Dockerfile.
- Эндпоинт `/health` заранее закладывается в приложение для последующего использования в проверках работоспособности.
- Проект с самого начала размещается в Git-репозитории на GitHub, готовом для настройки CI/CD в следующем уроке.

## Урок 10.3. Настройка CI: автотесты и сборка образа
**Время: 60 минут**

### Цель урока
Реализовать этап непрерывной интеграции итогового проекта: автоматические тесты и сборку Docker-образа.

### 1. Структура workflow-файла

```
devops-final-project/
├── .github/
│   └── workflows/
│       └── ci-cd.yml
├── app.py
├── test_app.py
├── requirements.txt
└── Dockerfile
```

### 2. Этап тестирования

```yaml
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      
      - name: Установка зависимостей
        run: pip install -r requirements.txt
      
      - name: Запуск тестов
        run: python -m unittest test_app.py
```

### 3. Добавление этапа сборки образа

```yaml
  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Вход в Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}
      
      - name: Сборка и публикация образа
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: ${{ secrets.DOCKERHUB_USERNAME }}/devops-final-app:latest
```

Обрати внимание на `needs: test` — этап сборки образа запустится только после успешного прохождения тестов, что соответствует принципу CI, изученному в Уроке 5.4.

### 4. Настройка секретов в GitHub

В настройках репозитория («Settings → Secrets and variables → Actions») необходимо добавить `DOCKERHUB_USERNAME` и `DOCKERHUB_TOKEN` — токен создаётся в настройках аккаунта Docker Hub («Account Settings → Security → New Access Token»).

### 5. Проверка результата

После push в ветку `main` пайплайн должен автоматически запуститься, выполнить тесты, а затем собрать и опубликовать Docker-образ. Прогресс можно отслеживать во вкладке «Actions» репозитория на GitHub.

### 6. Настройка Branch Protection Rule (по желанию)

Для более полного соответствия практикам, изученным в Уроке 5.4, можно настроить правило защиты ветки `main`, требующее успешного прохождения job `test` перед разрешением слияния Pull Request.

### Резюме урока

- Этап CI включает автоматический запуск тестов при каждом push и Pull Request, а сборка образа выполняется только после их успешного прохождения.
- Учётные данные для публикации образа в Docker Hub хранятся как защищённые секреты GitHub, а не в самом коде пайплайна.
- Полный CI-этап итогового проекта объединяет знания модулей 3 (Git), 4 (Docker) и 5 (CI/CD) в единый рабочий процесс.

## Урок 10.4. Настройка CD: автоматический деплой в облако/кластер
**Время: 65 минут**

### Цель урока
Завершить пайплайн итогового проекта этапом автоматического развёртывания приложения.

### 1. Выбор варианта развёртывания

В зависимости от доступных ресурсов и приоритетов, для итогового проекта можно выбрать один из вариантов:
- Развёртывание на облачном сервере через SSH (проще, требует облачный сервер, Модуль 6).
- Развёртывание в локальном Kubernetes-кластере через Minikube (сложнее, но глубже демонстрирует материал Модуля 7).

Ниже разобран вариант с развёртыванием на сервере через SSH как более доступный для большинства учащихся.

### 2. Подготовка сервера

На облачном сервере (Урок 6.4) должен быть установлен Docker, а также настроен и запущен через docker-compose пустой каркас проекта, готовый к обновлению:

```yaml
# docker-compose.yml на сервере
version: "3.9"

services:
  web:
    image: username/devops-final-app:latest
    ports:
      - "80:5000"
    restart: unless-stopped
```

### 3. Добавление этапа деплоя в пайплайн

```yaml
  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Деплой на сервер
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SERVER_SSH_KEY }}
          script: |
            cd /home/ubuntu/devops-final-project
            docker compose pull
            docker compose up -d
```

### 4. Настройка необходимых секретов

В GitHub Secrets добавляются `SERVER_HOST` (IP-адрес сервера), `SERVER_USER` (имя пользователя для SSH) и `SERVER_SSH_KEY` (приватный SSH-ключ для подключения, см. Урок 6.4).

### 5. Проверка полного цикла

После push в `main`:
1. Запускаются тесты.
2. Собирается и публикуется новый Docker-образ.
3. На сервере выполняется `docker compose pull` (скачивание нового образа) и `docker compose up -d` (перезапуск с новой версией).
4. Приложение обновляется автоматически, без ручного вмешательства.

### 6. Проверка работы приложения после деплоя

```bash
curl http://<IP_сервера>/health
```

### 7. Добавление проверки здоровья приложения после деплоя (опционально)

Для большей надёжности можно добавить в пайплайн дополнительный шаг, проверяющий, что приложение действительно ответило успешно после деплоя, и в противном случае — сигнализирующий об ошибке:

```yaml
      - name: Проверка работоспособности после деплоя
        run: |
          sleep 10
          curl -f http://${{ secrets.SERVER_HOST }}/health || exit 1
```

### Резюме урока

- Этап CD выполняется только после успешной сборки образа (`needs: build`) и автоматически обновляет приложение на сервере через SSH.
- Все данные для подключения к серверу хранятся как защищённые секреты GitHub, а не в открытом виде.
- Дополнительная проверка работоспособности после деплоя (health check) повышает надёжность автоматического процесса развёртывания.

## Урок 10.5. Презентация и защита проекта
**Время: 40 минут**

### Цель урока
Подготовить итоговый проект к демонстрации и защитить проделанную работу.

### 1. Финальная проверка перед презентацией

- Убедиться, что весь пайплайн (тесты → сборка → деплой) успешно проходит от начала до конца при push в `main`.
- Проверить, что приложение действительно доступно и работает после автоматического деплоя.
- Убедиться, что все секреты и учётные данные не попали в открытый код репозитория.
- Добавить `README.md` с описанием архитектуры и инструкцией по воспроизведению.

### 2. Пример README.md

```markdown
# DevOps Итоговый проект

Полный CI/CD пайплайн для простого Flask-приложения.

## Архитектура
Git push → GitHub Actions (тесты → сборка Docker-образа → публикация) 
→ автоматический деплой на сервер → мониторинг через Prometheus/Grafana

## Как воспроизвести
1. Настроить секреты в GitHub: DOCKERHUB_USERNAME, DOCKERHUB_TOKEN, SERVER_HOST, SERVER_USER, SERVER_SSH_KEY
2. Push в ветку main запустит полный пайплайн автоматически
```

### 3. Структура презентации проекта

1. **Архитектура.** Общая схема пайплайна и используемая инфраструктура (2 минуты).
2. **Демонстрация.** Живой пример: внести небольшое изменение в код, сделать push и показать, как оно автоматически проходит через весь пайплайн до развёртывания (3–4 минуты).
3. **Технические детали.** Какие инструменты и практики использованы из каждого модуля курса, какие были сложности (2–3 минуты).
4. **Возможные улучшения.** Что можно добавить при наличии больше времени: Kubernetes вместо простого сервера, Terraform для управления инфраструктурой, расширенный мониторинг с алертами (1–2 минуты).

### 4. Частые вопросы, к которым стоит подготовиться

- Что произойдёт, если тесты не пройдут — остановится ли деплой?
- Как организовано хранение и защита секретных данных в пайплайне?
- Как бы ты организовал откат (rollback) в случае неудачного деплоя?
- Как можно расширить проект для поддержки нескольких окружений (staging и production)?

### 5. Значение итогового проекта

Итоговый проект демонстрирует не разрозненные знания об отдельных инструментах, а понимание того, как Git, Docker, CI/CD, облачная инфраструктура и мониторинг работают вместе как единая система — именно это и составляет суть работы DevOps-инженера на практике.

### 6. Что дальше после курса

- **Kubernetes глубже.** Изучение продвинутых тем: Helm, Ingress, StatefulSets, операторы.
- **Terraform в реальных проектах.** Работа с remote state, модулями Terraform, множественными окружениями.
- **Сертификации.** Рассмотреть профильные сертификации облачных провайдеров (AWS Certified DevOps Engineer, Certified Kubernetes Administrator) для подтверждения квалификации.
- **SRE-практики.** Более глубокое изучение подходов к обеспечению надёжности систем (SLI/SLO/SLA, error budgets).

### Резюме урока

- Перед презентацией важно убедиться, что полный пайплайн работает end-to-end, и подготовить понятную документацию проекта.
- Презентация должна включать живую демонстрацию работы пайплайна, а не только описание архитектуры на словах.
- Итоговый проект показывает умение связать все изученные DevOps-практики в единый работающий процесс — ключевой навык для дальнейшей карьеры в этой области.
