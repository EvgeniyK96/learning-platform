# Модуль 8. Инфраструктура как код (IaC): Terraform

## Урок 8.1. Идея Infrastructure as Code
**Время: 35 минут**

### Цель урока
Понять концепцию Infrastructure as Code и её преимущества по сравнению с ручной настройкой инфраструктуры.

### 1. Проблема ручного управления инфраструктурой

Ручная настройка серверов, сетей и прочих облачных ресурсов через веб-консоль провайдера (Урок 6.4) работает для небольших экспериментов, но плохо масштабируется на реальные проекты:

- Сложно точно воспроизвести одно и то же окружение дважды (например, тестовое и продакшен).
- Отсутствует история изменений — непонятно, кто, когда и почему изменил конфигурацию сервера.
- Высокий риск человеческой ошибки при повторяющихся ручных действиях.
- Сложно быстро восстановить инфраструктуру после серьёзного сбоя.

### 2. Что такое Infrastructure as Code

Infrastructure as Code (IaC) — подход, при котором инфраструктура (серверы, сети, базы данных, права доступа) описывается в виде текстовых конфигурационных файлов, а не настраивается вручную через графический интерфейс. Эти файлы хранятся в системе контроля версий (Git), как и обычный код приложения.

### 3. Ключевые преимущества IaC

- **Воспроизводимость.** Одна и та же конфигурация гарантированно создаёт идентичную инфраструктуру в любой момент времени.
- **Версионирование.** Полная история изменений инфраструктуры хранится в Git, как и для обычного кода.
- **Совместная работа.** Изменения инфраструктуры можно проверять через Pull Request, как обычный код (Урок 3.5).
- **Быстрое восстановление.** При сбое инфраструктуру можно пересоздать заново из кода за минуты, а не настраивать вручную заново.
- **Документирование «по умолчанию».** Сам код служит актуальной документацией того, как устроена инфраструктура — не нужно поддерживать отдельные текстовые описания.

### 4. Декларативный vs императивный подход

- **Императивный подход** — пошаговое описание того, *как именно* создать нужное состояние («сначала создай сервер, потом установи ПО, потом настрой сеть»).
- **Декларативный подход** — описание *желаемого конечного результата*, а инструмент сам определяет, какие шаги нужны для его достижения («сервер с такими характеристиками должен существовать»).

Большинство современных IaC-инструментов, включая Terraform, следуют декларативному подходу.

### 5. Основные инструменты IaC

| Инструмент | Особенности |
|---|---|
| **Terraform** | универсальный, работает с большинством облачных провайдеров одновременно |
| **AWS CloudFormation** | специфичен только для AWS |
| **Ansible** | больше ориентирован на настройку уже существующих серверов (конфигурационный менеджмент), чем на создание инфраструктуры |
| **Pulumi** | похож на Terraform, но конфигурация пишется на обычных языках программирования (Python, JavaScript и др.) |

В этом модуле мы сосредоточимся на Terraform как наиболее универсальном и широко распространённом инструменте.

### 6. IaC как часть общего DevOps-цикла

Инфраструктура, описанная как код, легко встраивается в CI/CD пайплайны (Модуль 5): изменения инфраструктуры проходят через Pull Request и код-ревью, автоматически проверяются, и только после одобрения применяются к реальным облачным ресурсам — точно так же, как обычные изменения кода приложения.

### Резюме урока

- Infrastructure as Code описывает инфраструктуру в виде версionируемого кода вместо ручной настройки через веб-интерфейс.
- Ключевые преимущества: воспроизводимость, версионирование истории изменений, возможность код-ревью, быстрое восстановление после сбоев.
- Terraform — декларативный, независимый от конкретного облачного провайдера инструмент IaC, который мы изучим подробнее в этом модуле.

## Урок 8.2. Основы синтаксиса Terraform (HCL)
**Время: 50 минут**

### Цель урока
Изучить базовый синтаксис языка конфигурации Terraform — HCL (HashiCorp Configuration Language).

### 1. Установка Terraform (обзорно)

Terraform устанавливается как отдельная консольная утилита — инструкции для конкретной операционной системы доступны на официальном сайте terraform.io. Проверка установки:

```bash
terraform --version
```

### 2. Структура конфигурационного файла

Конфигурация Terraform пишется в файлах с расширением `.tf`, используя язык HCL:

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "eu-central-1"
}
```

### 3. Что такое provider

Provider (провайдер) — плагин Terraform, позволяющий взаимодействовать с конкретной платформой (AWS, Google Cloud, Azure, и даже такими сервисами, как GitHub или Docker). Provider определяет, какие типы ресурсов доступны для создания и управления.

### 4. Ресурсы (resource)

Ресурс — основной строительный блок конфигурации Terraform, описывающий конкретный объект инфраструктуры, который нужно создать:

```hcl
resource "aws_instance" "my_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  
  tags = {
    Name = "MyFirstServer"
  }
}
```

Синтаксис: `resource "тип_ресурса" "локальное_имя" { настройки }`. Локальное имя (`my_server`) используется только внутри конфигурации Terraform для ссылки на этот ресурс — оно не видно самому облачному провайдеру.

### 5. Переменные (variables)

Переменные позволяют избежать дублирования значений и делают конфигурацию более гибкой:

```hcl
variable "instance_type" {
  description = "Тип виртуальной машины"
  type        = string
  default     = "t2.micro"
}

resource "aws_instance" "my_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = var.instance_type
}
```

### 6. Выходные значения (outputs)

Outputs позволяют получить значения, сгенерированные при создании ресурсов (например, публичный IP-адрес созданного сервера), после применения конфигурации:

```hcl
output "server_ip" {
  value = aws_instance.my_server.public_ip
}
```

### 7. Комментарии в HCL

```hcl
# Однострочный комментарий
// Также однострочный комментарий

/*
Многострочный
комментарий
*/
```

### Резюме урока

- Конфигурация Terraform пишется на языке HCL в файлах с расширением `.tf`.
- `provider` подключает конкретную платформу (AWS, GCP и др.), `resource` описывает конкретный объект инфраструктуры для создания.
- Переменные (`variable`) делают конфигурацию гибкой и переиспользуемой, выходные значения (`output`) возвращают информацию о созданных ресурсах.

## Урок 8.3. Создание ресурсов через Terraform
**Время: 50 минут**

### Цель урока
Освоить базовый рабочий цикл Terraform: инициализация, планирование и применение конфигурации.

### 1. Основной рабочий цикл Terraform

Стандартный цикл работы с Terraform состоит из трёх шагов: `init` → `plan` → `apply`.

### 2. Инициализация: terraform init

```bash
terraform init
```

Эта команда скачивает необходимые плагины-провайдеры (например, для AWS) и подготавливает рабочую папку для работы с Terraform. Выполняется один раз при начале работы с новой конфигурацией (и после добавления новых провайдеров).

### 3. Планирование изменений: terraform plan

```bash
terraform plan
```

Команда `plan` показывает, какие именно изменения Terraform собирается внести в реальную инфраструктуру, **не применяя их фактически**. Это критически важный шаг — он позволяет проверить, что Terraform собирается сделать именно то, что задумано, прежде чем реально изменить облачные ресурсы (например, случайно не удалить существующий важный сервер).

Вывод `plan` использует условные обозначения: `+` (будет создано), `-` (будет удалено), `~` (будет изменено).

### 4. Применение конфигурации: terraform apply

```bash
terraform apply
```

После проверки плана и подтверждения (`yes`) Terraform фактически создаёт, изменяет или удаляет ресурсы в реальной инфраструктуре согласно конфигурации.

### 5. Файл состояния (state)

После применения конфигурации Terraform сохраняет файл состояния (`terraform.tfstate`), в котором хранится информация о том, какие ресурсы реально существуют и в каком состоянии. Этот файл использует Terraform при последующих запусках `plan`/`apply`, чтобы понять, что уже создано, а что нужно ещё создать или изменить.

> Важно: файл состояния может содержать чувствительную информацию и не должен обычно попадать в публичный Git-репозиторий; в командной работе для его хранения часто используют удалённое, защищённое хранилище (remote state), например S3-бакет с блокировкой через DynamoDB.

### 6. Удаление созданной инфраструктуры

```bash
terraform destroy
```

Команда полностью удаляет все ресурсы, описанные и созданные текущей конфигурацией — крайне полезно для учебных экспериментов, чтобы не платить за забытые облачные ресурсы, но требует особой осторожности в реальных проектах.

### 7. Практический пример: полный цикл создания S3-бакета (AWS)

```hcl
provider "aws" {
  region = "eu-central-1"
}

resource "aws_s3_bucket" "my_bucket" {
  bucket = "my-unique-bucket-name-12345"
}

output "bucket_name" {
  value = aws_s3_bucket.my_bucket.bucket
}
```

```bash
terraform init
terraform plan
terraform apply
# ... проверка результата ...
terraform destroy
```

### Резюме урока

- Основной цикл работы с Terraform: `init` (инициализация) → `plan` (просмотр планируемых изменений) → `apply` (применение).
- `terraform plan` — критически важный шаг для проверки планируемых изменений перед их фактическим применением к реальной инфраструктуре.
- Файл состояния (`terraform.tfstate`) хранит информацию о реально созданных ресурсах и не должен публично храниться в открытом виде.

## Урок 8.4. Практика: описание простой инфраструктуры кодом
**Время: 50 минут**

### Цель урока
Закрепить материал модуля, описав через Terraform небольшую, но полноценную инфраструктуру.

### 1. Постановка задачи

Опишем через Terraform инфраструктуру для размещения простого веб-приложения: виртуальный сервер и правило сети (security group), разрешающее доступ по HTTP и SSH — аналогично тому, что настраивалось вручную в Уроке 6.4, но теперь полностью через код.

### 2. Структура проекта

```
terraform_project/
├── main.tf         # основная конфигурация
├── variables.tf     # объявление переменных
└── outputs.tf        # выходные значения
```

### 3. Файл variables.tf

```hcl
variable "region" {
  description = "Регион AWS"
  type        = string
  default     = "eu-central-1"
}

variable "instance_type" {
  description = "Тип виртуальной машины"
  type        = string
  default     = "t2.micro"
}
```

### 4. Файл main.tf

```hcl
provider "aws" {
  region = var.region
}

resource "aws_security_group" "web_sg" {
  name        = "web-sg"
  description = "Разрешить HTTP и SSH"
  
  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

resource "aws_instance" "web_server" {
  ami                    = "ami-0c55b159cbfafe1f0"
  instance_type           = var.instance_type
  vpc_security_group_ids  = [aws_security_group.web_sg.id]
  
  tags = {
    Name = "WebServer"
  }
}
```

Обрати внимание на строку `vpc_security_group_ids = [aws_security_group.web_sg.id]` — это пример **ссылки между ресурсами**: Terraform автоматически определяет, что security group должна быть создана раньше сервера, который на неё ссылается, без необходимости указывать порядок вручную.

### 5. Файл outputs.tf

```hcl
output "server_public_ip" {
  description = "Публичный IP-адрес созданного сервера"
  value       = aws_instance.web_server.public_ip
}
```

### 6. Применение конфигурации

```bash
terraform init
terraform plan
terraform apply
```

После успешного применения Terraform выведет публичный IP-адрес созданного сервера, определённый в `outputs.tf`.

### 7. Важное напоминание об учебных ресурсах

После завершения практики обязательно вызови `terraform destroy`, чтобы удалить созданные облачные ресурсы и не платить за них дальше — особенно если используется не бесплатный уровень провайдера.

```bash
terraform destroy
```

### Резюме урока

- Реальные Terraform-проекты обычно разделяются на несколько файлов: конфигурация ресурсов, переменные, выходные значения.
- Terraform автоматически определяет порядок создания связанных ресурсов на основе ссылок между ними в коде.
- Всегда важно вызывать `terraform destroy` для учебных или временных ресурсов, чтобы избежать неожиданных расходов на облачную инфраструктуру.
