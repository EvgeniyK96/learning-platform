## Урок 1.1. Проблема «у меня работает» и идея контейнеров
**Время: 35 минут**

### Цель урока
Понять проблему, которую решает контейнеризация, и получить общее представление о принципах работы Docker.

### 1. Классическая проблема разработки

Ситуация, знакомая практически каждому разработчику: программа отлично работает на локальном компьютере, но «падает» при развёртывании на сервере или на компьютере коллеги. Причины могут быть разными: другая версия языка программирования, отсутствующие зависимости, отличающиеся настройки операционной системы. Эта проблема получила ироничное неформальное название «у меня работает» (it works on my machine).

### 2. Как эту проблему решали раньше

- **Подробная документация по настройке окружения.** Ненадёжно: документация быстро устаревает, а ручная настройка легко приводит к ошибкам и упущениям.
- **Виртуальные машины.** Полноценная эмуляция целой операционной системы для каждого приложения — надёжно, но очень «тяжело» с точки зрения потребления ресурсов, размера и времени запуска.

### 3. Идея контейнеризации

Контейнер — изолированная среда выполнения, содержащая приложение вместе со всеми необходимыми ему зависимостями (библиотеки, настройки, переменные окружения), но при этом использующая общее ядро операционной системы хоста, а не эмулирующая целиком отдельную ОС, как виртуальная машина. Это делает контейнеры значительно более лёгкими и быстрыми, чем виртуальные машины, сохраняя при этом изоляцию окружения приложения.

### 4. Контейнеры vs виртуальные машины

| | Виртуальная машина | Контейнер |
|---|---|---|
| Что виртуализируется | Целая операционная система | Только приложение и его окружение |
| Типичный размер | Гигабайты | Обычно десятки–сотни мегабайт |
| Время запуска | Минуты | Секунды |
| Изоляция | Полная (отдельное ядро ОС) | На уровне процессов (общее ядро ОС хоста) |

### 5. Что такое Docker

Docker — самая популярная в мире платформа для создания, запуска и управления контейнерами. Docker позволяет один раз описать окружение приложения (в специальном файле — Dockerfile, Урок 2.2) и затем запускать абсолютно идентичное окружение на любой машине, где установлен Docker: будь то ноутбук разработчика, тестовый сервер или продакшен-сервер в облаке.

### 6. Ключевые преимущества контейнеризации

- **Воспроизводимость.** Приложение ведёт себя одинаково в любом окружении, где запущено через Docker.
- **Изоляция.** Разные приложения на одном сервере не конфликтуют друг с другом и своими зависимостями.
- **Быстрое масштабирование.** Запуск дополнительных копий контейнера занимает секунды.
- **Упрощение процессов разработки и доставки.** Приложение и его окружение упаковываются в единый переносимый артефакт, используемый на всех этапах — от разработки до продакшена.

### 7. Где применяется Docker сегодня

Docker стал фактическим стандартом индустрии для упаковки и развёртывания приложений — от небольших личных проектов до крупнейших технологических компаний. Docker лежит в основе большинства современных DevOps-практик, включая CI/CD пайплайны и системы оркестрации вроде Kubernetes.

### Резюме урока

- Проблема «у меня работает» возникает из-за расхождений между окружением разработчика и окружением, где запускается приложение (сервер, окружение коллеги).
- Контейнеризация решает эту проблему, упаковывая приложение вместе со всеми зависимостями в переносимую изолированную среду.
- Контейнеры значительно легче и быстрее виртуальных машин, так как используют общее ядро операционной системы хоста вместо полной эмуляции отдельной ОС.

## Урок 1.2. Установка Docker и первый запуск контейнера (docker run)
**Время: 40 минут**

### Цель урока
Установить Docker и запустить свой первый контейнер.

### 1. Установка Docker

**Windows/macOS:** скачать и установить Docker Desktop с официального сайта docker.com — этот единый пакет включает всё необходимое для работы с Docker, включая удобный графический интерфейс.

**Linux (Ubuntu):**
```bash
sudo apt update
sudo apt install docker.io
sudo systemctl start docker
sudo systemctl enable docker
```

### 2. Проверка установки

```bash
docker --version
docker run hello-world
```

Команда `docker run hello-world` скачивает небольшой тестовый образ и запускает из него контейнер — успешный вывод приветственного сообщения подтверждает, что Docker установлен и работает корректно.

### 3. Разбор команды docker run

`docker run` — основная команда для создания и запуска контейнера на основе указанного образа:

```bash
docker run nginx
```

Если образ `nginx` ещё не скачан локально, Docker автоматически скачает его из реестра (по умолчанию — Docker Hub), прежде чем запустить контейнер.

### 4. Публикация портов: флаг -p

Чтобы получить доступ к приложению внутри контейнера снаружи (например, из браузера на компьютере хоста), необходимо «пробросить» порт с помощью флага `-p`:

```bash
docker run -p 8080:80 nginx
```

Эта команда связывает порт 8080 на компьютере хоста с портом 80 внутри контейнера (стандартный порт для веб-сервера nginx). Формат записи: `порт_хоста:порт_контейнера`. После этого приложение станет доступно по адресу `http://localhost:8080`.

### 5. Запуск в фоновом режиме: флаг -d

```bash
docker run -d -p 8080:80 nginx
```

Флаг `-d` (сокращение от detached, «отсоединённый») запускает контейнер в фоновом режиме, освобождая терминал для дальнейшей работы, вместо блокировки текущей сессии выводом логов контейнера.

### 6. Именование контейнеров: флаг --name

Для удобства управления можно дать контейнеру понятное имя вместо использования автоматически сгенерированного случайного идентификатора:

```bash
docker run -d -p 8080:80 --name my_website nginx
```

### 7. Практический пример: запуск веб-сервера

```bash
docker run -d -p 8080:80 --name my_first_container nginx
```

После выполнения этой команды в браузере по адресу `http://localhost:8080` должна открыться приветственная страница веб-сервера nginx, работающего внутри Docker-контейнера.

### Резюме урока

- Docker Desktop — рекомендуемый способ установки на Windows и macOS, на Linux Docker устанавливается через штатный менеджер пакетов дистрибутива.
- `docker run` создаёт и запускает контейнер на основе указанного образа, автоматически скачивая его при необходимости.
- Флаг `-p порт_хоста:порт_контейнера` пробрасывает порт для внешнего доступа к приложению, флаг `-d` запускает контейнер в фоновом режиме.

## Урок 1.3. Основные команды: ps, stop, rm, logs, exec
**Время: 35 минут**

### Цель урока
Освоить базовый набор команд для управления жизненным циклом контейнеров.

### 1. Просмотр запущенных контейнеров: docker ps

```bash
docker ps         # список только ЗАПУЩЕННЫХ контейнеров
docker ps -a        # список ВСЕХ контейнеров, включая остановленные
```

Вывод включает идентификатор контейнера, использованный образ, время запуска, проброшенные порты и имя контейнера.

### 2. Остановка контейнера: docker stop

```bash
docker stop my_first_container      # остановка по имени
docker stop a1b2c3d4                   # остановка по идентификатору (или его началу)
```

`docker stop` посылает контейнеру сигнал на корректное завершение работы, давая приложению внутри время для «мягкой» остановки (сохранения данных, закрытия соединений), прежде чем контейнер будет остановлен принудительно, если он не отреагирует в течение определённого времени.

### 3. Повторный запуск остановленного контейнера: docker start

```bash
docker start my_first_container
```

В отличие от `docker run` (который создаёт новый контейнер), `docker start` запускает уже существующий, ранее остановленный контейнер.

### 4. Удаление контейнера: docker rm

```bash
docker rm my_first_container      # удаление остановленного контейнера
docker rm -f my_first_container   # принудительная остановка И удаление одной командой
```

Обычный `docker rm` работает только для уже остановленных контейнеров — попытка удалить запущенный контейнер без флага `-f` завершится ошибкой.

### 5. Просмотр логов контейнера: docker logs

```bash
docker logs my_first_container       # показать все накопленные логи контейнера
docker logs -f my_first_container    # непрерывный просмотр логов в реальном времени (аналог tail -f)
```

Просмотр логов — один из основных инструментов диагностики проблем при работе с контейнерами, особенно когда приложение внутри контейнера ведёт себя не так, как ожидается.

### 6. Выполнение команды внутри работающего контейнера: docker exec

```bash
docker exec my_first_container ls /usr/share/nginx/html   # выполнить одну команду внутри контейнера

docker exec -it my_first_container /bin/bash                  # открыть интерактивный терминал ВНУТРИ контейнера
```

Флаги `-it` (interactive + tty) вместе обеспечивают полноценную интерактивную сессию терминала — эта комбинация часто используется для «захода» внутрь работающего контейнера с целью отладки или проверки его содержимого напрямую.

### 7. Практический пример: полный цикл управления контейнером

```bash
docker run -d -p 8080:80 --name web nginx
docker ps                                  # проверяем, что контейнер запущен
docker logs web                          # смотрим логи
docker exec -it web /bin/bash           # заходим внутрь для проверки
exit                                          # выходим из внутреннего терминала контейнера
docker stop web                          # останавливаем контейнер
docker rm web                             # удаляем контейнер
```

### Резюме урока

- `docker ps`/`docker ps -a` показывают запущенные и все контейнеры соответственно, `docker stop`/`docker start` управляют их работой.
- `docker rm` удаляет контейнер (обязательно предварительно остановленный, если не используется флаг `-f`).
- `docker logs` — основной инструмент просмотра вывода контейнера, `docker exec -it ... /bin/bash` позволяет «зайти» внутрь работающего контейнера для отладки.

## Урок 2.1. Docker-образы: что это, работа с Docker Hub
**Время: 40 минут**

### Цель урока
Понять разницу между образом и контейнером и научиться работать с публичным реестром образов Docker Hub.

### 1. Разница между образом и контейнером

- **Образ (image)** — неизменяемый «шаблон» или «чертёж», содержащий приложение и все его зависимости.
- **Контейнер (container)** — запущенный, «живой» экземпляр образа, который можно запускать, останавливать, удалять.

Полезная аналогия из программирования: образ похож на класс, контейнер — на конкретный объект, созданный на его основе. От одного образа можно одновременно запустить сколько угодно независимых контейнеров.

### 2. Просмотр загруженных образов

```bash
docker images
```

Вывод включает название образа, тег (версию), идентификатор и размер занимаемого места на диске.

### 3. Загрузка образа без немедленного запуска: docker pull

```bash
docker pull python:3.12
docker pull node:20
docker pull postgres:16
```

`docker pull` только скачивает образ, не создавая и не запуская из него контейнер, — эта команда полезна, если образ нужно заранее подготовить для последующего использования.

### 4. Что такое Docker Hub

Docker Hub (hub.docker.com) — крупнейший в мире публичный реестр Docker-образов. На нём хранятся официальные образы популярного программного обеспечения (Python, Node.js, PostgreSQL, nginx и многие другие), а также образы, опубликованные независимыми разработчиками и организациями.

### 5. Теги образов (версии)

Большинство образов на Docker Hub имеют несколько тегов, соответствующих разным версиям и вариантам сборки:

```bash
docker pull python:3.12          # конкретная зафиксированная версия
docker pull python:latest          # последняя доступная версия (используется по умолчанию, если тег не указан)
docker pull python:3.12-slim      # облегчённая версия образа с минимальным набором компонентов
```

Использование конкретного версионного тега (например, `python:3.12`) вместо `latest` — рекомендуемая практика для продакшен-окружений, так как гарантирует предсказуемое поведение приложения при последующих обновлениях образа.

### 6. Удаление образов

```bash
docker rmi python:3.12       # удалить конкретный образ
docker image prune              # удалить все образы, не используемые ни одним контейнером
```

### 7. Практический пример: сравнение размеров образов

```bash
docker pull python:3.12
docker pull python:3.12-slim
docker images | grep python
```

Эта команда наглядно продемонстрирует, что «slim»-версия образа занимает значительно меньше места, чем полная версия — полезное знание при оптимизации размера итоговых образов приложений.

### Резюме урока

- Образ — неизменяемый шаблон приложения, контейнер — запущенный на его основе экземпляр; от одного образа можно запустить множество независимых контейнеров.
- Docker Hub — крупнейший публичный реестр образов, откуда образы скачиваются командой `docker pull`.
- Использование конкретных версионных тегов образов (а не `latest`) — рекомендуемая практика для предсказуемости продакшен-окружений.

## Урок 2.2. Написание собственного Dockerfile
**Время: 55 минут**

### Цель урока
Научиться описывать собственный Docker-образ с помощью Dockerfile.

### 1. Что такое Dockerfile

Dockerfile — текстовый файл с набором последовательных инструкций, описывающих, как именно нужно собрать Docker-образ: с какого базового образа начать, какие файлы скопировать внутрь, какие команды выполнить в процессе сборки, что запускать при старте контейнера.

### 2. Базовая структура Dockerfile

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install -r requirements.txt --no-cache-dir

COPY . .

CMD ["python", "app.py"]
```

### 3. Разбор основных инструкций

| Инструкция | Назначение |
|---|---|
| `FROM` | указывает базовый образ, на основе которого строится новый |
| `WORKDIR` | задаёт рабочую директорию внутри создаваемого образа |
| `COPY` | копирует файлы с локальной машины внутрь образа |
| `RUN` | выполняет команду во время сборки образа (например, установку зависимостей) |
| `CMD` | команда, выполняемая при запуске контейнера (может быть указана только одна на весь образ) |
| `EXPOSE` | документирует, какой порт использует приложение внутри контейнера |
| `ENV` | задаёт переменную окружения внутри образа |

### 4. Сборка образа: docker build

```bash
docker build -t my_app:1.0 .
```

Флаг `-t` («tag») задаёт имя и тег собираемого образа, точка `.` в конце команды указывает контекст сборки — обычно текущую папку, где находится сам файл Dockerfile и весь исходный код приложения.

### 5. Запуск собственного образа

```bash
docker run -d -p 5000:5000 --name my_app_container my_app:1.0
```

### 6. Кэширование слоёв сборки

Docker кэширует результат выполнения каждой инструкции Dockerfile как отдельный «слой». Если содержимое, влияющее на конкретный слой (например, файл `requirements.txt`), не изменилось с прошлой сборки, Docker переиспользует уже готовый слой из кэша вместо повторного выполнения, что значительно ускоряет процесс повторной сборки образа.

Именно поэтому в примере из пункта 2 сначала копируется файл зависимостей (`requirements.txt`) и устанавливаются сами зависимости, и только затем копируется весь остальной код приложения: изменение кода приложения не приводит к повторной, более медленной установке всех зависимостей при каждой пересборке.

### 7. Пример Dockerfile для Node.js-приложения

```dockerfile
FROM node:20-slim

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .

EXPOSE 3000
CMD ["node", "server.js"]
```

### 8. Файл .dockerignore

Аналогично `.gitignore` в Git, файл `.dockerignore` позволяет исключить ненужные файлы и папки (временные файлы, локальные зависимости, файлы с секретами) из контекста сборки Docker-образа, ускоряя сборку и уменьшая итоговый размер образа:

```
node_modules
.git
*.log
.env
```

### Резюме урока

- Dockerfile описывает пошаговую инструкцию сборки собственного Docker-образа: базовый образ, копирование файлов, установка зависимостей, команда запуска.
- `docker build -t имя:тег .` собирает образ на основе Dockerfile из указанного контекста (обычно текущей папки).
- Правильный порядок инструкций (сначала зависимости, затем код) ускоряет повторную сборку за счёт механизма кэширования слоёв Docker.

## Урок 2.3. Тома (volumes) и сети (networks) в Docker
**Время: 40 минут**

### Цель урока
Изучить механизмы сохранения данных вне контейнера и организации сетевого взаимодействия между контейнерами.

### 1. Проблема временности данных в контейнере

По умолчанию все данные, созданные или изменённые внутри контейнера, безвозвратно теряются при его удалении — это неприемлемо, например, для баз данных, где важно сохранять накопленные данные между перезапусками и пересозданиями контейнера.

### 2. Что такое том (volume)

Том — механизм Docker для сохранения данных вне жизненного цикла конкретного контейнера. Данные, хранящиеся в томе, продолжают существовать даже после удаления использовавшего их контейнера и могут быть подключены к новому контейнеру.

### 3. Создание и использование именованного тома

```bash
docker volume create my_data                            # создать именованный том явно
docker run -d -v my_data:/var/lib/postgresql/data postgres:16   # подключить том к контейнеру
```

Формат флага `-v`: `имя_тома:путь_внутри_контейнера`. Если указанный том ещё не существует, Docker создаст его автоматически при первом использовании.

### 4. Просмотр и удаление томов

```bash
docker volume ls                  # список всех томов
docker volume rm my_data          # удаление конкретного тома
docker volume prune                 # удаление всех неиспользуемых томов
```

### 5. Bind mount — альтернативный способ монтирования

Помимо именованных томов, Docker поддерживает прямое монтирование конкретной папки с компьютера хоста внутрь контейнера (bind mount) — часто используется в разработке, чтобы изменения кода на хосте сразу отражались внутри контейнера без пересборки образа:

```bash
docker run -v /home/user/my_project:/app my_app:1.0
```

### 6. Сети Docker

По умолчанию каждый запущенный контейнер изолирован в сетевом плане от других контейнеров, если явно не настроено иное. Docker позволяет создавать собственные пользовательские сети, объединяющие несколько контейнеров, дающие им возможность обращаться друг к другу по имени контейнера, а не по IP-адресу:

```bash
docker network create my_network

docker run -d --name db --network my_network postgres:16
docker run -d --name app --network my_network my_app:1.0
```

Теперь контейнер `app` может обращаться к базе данных просто по имени `db` (например, `db:5432`), без необходимости знать её фактический внутренний IP-адрес, который может изменяться при пересоздании контейнера.

### 7. Просмотр сетей

```bash
docker network ls
docker network inspect my_network
```

### Резюме урока

- Тома (volumes) сохраняют данные вне жизненного цикла контейнера — необходимы для баз данных и любых данных, которые должны переживать пересоздание контейнера.
- Bind mount монтирует конкретную папку хоста внутрь контейнера — часто применяется в разработке для мгновенного отражения изменений кода.
- Пользовательские сети Docker позволяют контейнерам обращаться друг к другу по имени, а не по изменяющемуся IP-адресу.

## Урок 2.4. docker-compose: запуск многоконтейнерных приложений
**Время: 55 минут**

### Цель урока
Освоить docker-compose — инструмент для описания и одновременного запуска нескольких связанных контейнеров.

### 1. Проблема, которую решает docker-compose

Реальные приложения редко состоят из одного контейнера — типичный веб-сервис включает как минимум приложение и базу данных, часто также кэш и дополнительные вспомогательные сервисы. Запускать и настраивать каждый контейнер отдельными длинными командами `docker run` (с портами, томами, сетями) неудобно и подвержено ошибкам.

`docker-compose` позволяет описать все контейнеры проекта и их взаимные настройки в едином YAML-файле, после чего запускать (и останавливать) их все одной командой.

### 2. Базовая структура docker-compose.yml

```yaml
version: "3.9"

services:
  web:
    build: .
    ports:
      - "5000:5000"
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - db_data:/var/lib/postgresql/data

volumes:
  db_data:
```

### 3. Разбор структуры файла

- **services** — список сервисов (контейнеров), из которых состоит приложение.
- **build** — указывает, что образ для этого сервиса нужно собрать из локального Dockerfile (в отличие от `image`, скачивающего готовый образ из реестра).
- **depends_on** — задаёт порядок запуска сервисов (в примере `web` запустится после `db`, хотя это не гарантирует полной готовности базы данных к моменту старта приложения).
- **volumes** (внутри сервиса) — подключение тома для сохранения данных.
- **volumes** (на верхнем уровне файла) — объявление именованных томов, используемых сервисами.

### 4. Основные команды docker-compose

```bash
docker compose up            # запустить все описанные сервисы (вывод логов в текущий терминал)
docker compose up -d           # запустить все сервисы в фоновом режиме
docker compose down            # остановить и удалить все контейнеры проекта
docker compose ps                # список запущенных сервисов текущего проекта
docker compose logs web          # просмотр логов конкретного сервиса
docker compose logs -f web        # непрерывный просмотр логов в реальном времени
```

### 5. Автоматическая сеть между сервисами

По умолчанию `docker-compose` автоматически создаёт общую сеть для всех сервисов, описанных в одном файле — они могут обращаться друг к другу просто по имени сервиса (как описано в предыдущем уроке про пользовательские сети), без какой-либо дополнительной настройки.

### 6. Пересборка образов при изменении Dockerfile

```bash
docker compose up -d --build   # принудительная пересборка образов перед запуском
```

Флаг `--build` полезен, когда были внесены изменения в Dockerfile или в файлы, копируемые в образ, чтобы гарантировать использование актуальной версии образа, а не устаревшей закэшированной.

### 7. Практический пример: полное описание веб-приложения с базой данных

```yaml
version: "3.9"

services:
  web:
    build: .
    ports:
      - "5000:5000"
    environment:
      DATABASE_URL: postgresql://user:secret@db:5432/mydb
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: mydb
    volumes:
      - db_data:/var/lib/postgresql/data

volumes:
  db_data:
```

```bash
docker compose up -d
docker compose ps
docker compose logs web
```

### Резюме урока

- `docker-compose` описывает несколько связанных контейнеров в едином YAML-файле и позволяет управлять ими всеми одной командой.
- Сервисы в одном compose-файле автоматически объединены общей сетью и могут обращаться друг к другу по имени сервиса.
- Флаг `--build` гарантирует пересборку образов перед запуском при внесении изменений в Dockerfile или связанные файлы.
