## Урок 10.1. Планирование проекта и постановка задачи
**Время: 40 минут**

### Цель урока
Научиться правильно формулировать задачу перед началом разработки и составить техническое задание для собственного итогового проекта.

### 1. Зачем нужно планирование

Одна из самых частых ошибок начинающих разработчиков — сразу садиться писать код, не продумав заранее, что именно должна делать программа. Это приводит к постоянным переделкам и потере времени. Даже небольшой проект стоит начинать с краткого плана.

### 2. Выбор темы итогового проекта

Примеры тем, подходящих для итогового проекта на основе материала курса:

- **Трекер задач (To-Do List)** — с сохранением данных в файл или базу данных SQLite.
- **Записная книжка контактов** — с добавлением, поиском, редактированием и удалением записей.
- **Простой учёт финансов** — доходы/расходы с категориями и итоговой статистикой.
- **Консольная игра** («Виселица», «Крестики-нолики», «Угадай число» с расширенной логикой).
- **Парсер данных** — сбор информации через API и сохранение её в файл/базу данных.
- **Приложение по мотивам Модуля 7** — расширение примера «Автопарк» или аналогичная предметная область (учёт книг в библиотеке, учёт студентов и т.д.).

### 3. Структура простого технического задания (ТЗ)

Хорошее ТЗ для учебного проекта отвечает на следующие вопросы:

1. **Какую задачу решает программа?** (одно-два предложения)
2. **Кто пользователь программы?** (например, «студент, который хочет отслеживать свои задачи»)
3. **Какие функции должны быть реализованы?** (список конкретных возможностей)
4. **Какие модули курса будут использованы?** (циклы, функции, ООП, файлы, база данных и т.д.)
5. **Как будут храниться данные?** (в переменных на время работы программы, в файле, в базе данных SQLite)

### 4. Пример технического задания

**Проект:** Консольный трекер задач

**Задача:** Помочь пользователю вести список дел с возможностью отмечать выполненные задачи.

**Функции:**
- Добавление новой задачи с названием и приоритетом.
- Просмотр списка всех задач.
- Отметка задачи как выполненной.
- Удаление задачи.
- Сохранение списка задач в файл, чтобы данные не терялись при перезапуске программы.

**Используемые темы курса:** функции, списки словарей, работа с файлами (JSON), обработка исключений, (по желанию) ООП.

**Хранение данных:** JSON-файл `tasks.json`.

### 5. Оценка объёма задачи

Для итогового проекта курса не стоит замахиваться на слишком масштабную идею. Хороший ориентир — проект, который можно реализовать за 3–5 часов работы (время модуля 10) и в котором явно используются минимум 3–4 темы, пройденные в курсе.

### 6. Домашнее задание к уроку

Составь техническое задание для своего итогового проекта по образцу из пункта 4: опиши задачу, пользователя, список функций, используемые темы курса и способ хранения данных.

### Резюме урока

- Планирование перед началом кодирования экономит значительно больше времени, чем занимает само составление плана.
- Хорошее техническое задание отвечает на вопросы: что делает программа, для кого, какие функции нужны и как хранятся данные.
- Итоговый проект должен быть выполнимым по объёму и явно опираться на темы, пройденные в курсе.

## Урок 10.2. Проектирование структуры приложения
**Время: 60 минут**

### Цель урока
Научиться проектировать структуру кода будущего приложения до начала написания самой логики: файлы, функции/классы и их взаимодействие.

### 1. От идеи к структуре

После того как задача сформулирована (Урок 10.1), следующий шаг — продумать, из каких частей будет состоять программа и как эти части будут взаимодействовать друг с другом.

### 2. Разбиение на файлы

Даже небольшой проект стоит разделить хотя бы на 2–3 файла:

```
task_tracker/
│
├── main.py            # точка входа, меню и основной цикл программы
├── tasks.py            # функции для работы с задачами (добавление, удаление и т.д.)
└── storage.py          # функции для сохранения/загрузки данных из файла
```

### 3. Проектирование функций до написания кода

Полезная практика — сначала описать «заголовки» будущих функций с кратким комментарием, что они должны делать, и только потом реализовывать их:

```python
# tasks.py

def add_task(tasks, title, priority):
    """Добавляет новую задачу в список задач"""
    pass

def complete_task(tasks, task_id):
    """Отмечает задачу с указанным id как выполненную"""
    pass

def remove_task(tasks, task_id):
    """Удаляет задачу с указанным id из списка"""
    pass

def show_tasks(tasks):
    """Выводит список всех задач на экран"""
    pass
```

Такой подход называется «заглушками» (stubs) — это позволяет сначала продумать общую архитектуру, а затем поэтапно наполнять функции реальной логикой.

### 4. Выбор структуры данных для хранения информации

Для трекера задач хорошим выбором будет список словарей:

```python
tasks = [
    {"id": 1, "title": "Сделать домашнее задание", "priority": "высокий", "done": False},
    {"id": 2, "title": "Купить продукты", "priority": "низкий", "done": False}
]
```

### 5. Проектирование пользовательского меню

Для консольного приложения полезно заранее продумать структуру меню:

```
1. Показать все задачи
2. Добавить задачу
3. Отметить задачу выполненной
4. Удалить задачу
5. Выйти
```

### 6. Простая диаграмма взаимодействия

```
main.py (меню, ввод пользователя)
   │
   ├──> tasks.py (логика работы с задачами)
   │
   └──> storage.py (сохранение/загрузка из tasks.json)
```

Такая схема помогает заранее увидеть, какие функции должны быть в каждом файле, и в каком порядке они будут вызываться.

### Резюме урока

- Прежде чем писать код, полезно спроектировать структуру файлов и функций «на бумаге» или в виде заглушек.
- Разделение проекта на несколько файлов (интерфейс, логика, хранение данных) делает код более организованным.
- Список словарей — удобная структура данных для хранения однотипных записей (задач, контактов, товаров и т.д.).

## Урок 10.3. Реализация основного функционала
**Время: 90 минут**

### Цель урока
Реализовать рабочую версию итогового проекта на основе технического задания (Урок 10.1) и структуры (Урок 10.2).

### 1. С чего начать реализацию

Рекомендуемый порядок разработки:

1. Реализовать функции хранения данных (сохранение/загрузка) — без этого невозможно проверить остальной функционал.
2. Реализовать основные функции логики (добавление, удаление, изменение).
3. Собрать всё вместе в главном файле с меню и пользовательским вводом.
4. Проверять работу программы после реализации каждой отдельной функции, а не в самом конце.

### 2. Пример реализации хранения данных (storage.py)

```python
import json
import os

FILENAME = "tasks.json"

def load_tasks():
    if not os.path.exists(FILENAME):
        return []
    with open(FILENAME, "r", encoding="utf-8") as file:
        return json.load(file)

def save_tasks(tasks):
    with open(FILENAME, "w", encoding="utf-8") as file:
        json.dump(tasks, file, ensure_ascii=False, indent=4)
```

### 3. Пример реализации логики (tasks.py)

```python
def add_task(tasks, title, priority):
    new_id = max([task["id"] for task in tasks], default=0) + 1
    tasks.append({
        "id": new_id,
        "title": title,
        "priority": priority,
        "done": False
    })
    return tasks

def complete_task(tasks, task_id):
    for task in tasks:
        if task["id"] == task_id:
            task["done"] = True
            return True
    return False

def remove_task(tasks, task_id):
    for task in tasks:
        if task["id"] == task_id:
            tasks.remove(task)
            return True
    return False

def show_tasks(tasks):
    if not tasks:
        print("Список задач пуст")
        return
    for task in tasks:
        status = "✓" if task["done"] else " "
        print(f"[{status}] {task['id']}. {task['title']} (приоритет: {task['priority']})")
```

### 4. Пример основного файла (main.py)

```python
from tasks import add_task, complete_task, remove_task, show_tasks
from storage import load_tasks, save_tasks

def main():
    tasks = load_tasks()

    while True:
        print("\n1. Показать задачи\n2. Добавить задачу\n3. Выполнить задачу\n4. Удалить задачу\n5. Выйти")
        choice = input("Выбери действие: ")

        if choice == "1":
            show_tasks(tasks)
        elif choice == "2":
            title = input("Название задачи: ")
            priority = input("Приоритет (высокий/средний/низкий): ")
            tasks = add_task(tasks, title, priority)
            save_tasks(tasks)
        elif choice == "3":
            task_id = int(input("ID задачи для выполнения: "))
            if complete_task(tasks, task_id):
                save_tasks(tasks)
                print("Задача отмечена как выполненная")
            else:
                print("Задача не найдена")
        elif choice == "4":
            task_id = int(input("ID задачи для удаления: "))
            if remove_task(tasks, task_id):
                save_tasks(tasks)
                print("Задача удалена")
            else:
                print("Задача не найдена")
        elif choice == "5":
            print("До встречи!")
            break
        else:
            print("Некорректный выбор")

if __name__ == "__main__":
    main()
```

### 5. Постепенная проверка каждой функции

После написания каждой функции стоит сразу проверить её отдельно, прежде чем переходить к следующей — это значительно упрощает поиск ошибок по сравнению с проверкой всего приложения целиком в самом конце.

### 6. Обработка некорректного ввода

На этом этапе стоит добавить базовую защиту от ошибок пользовательского ввода — например, обернуть преобразование `int(input(...))` в `try/except`, чтобы программа не «падала» при вводе текста вместо числа.

### Резюме урока

- Реализацию проекта стоит начинать с функций хранения данных, а не с интерфейса.
- Логику работы с данными и интерфейс пользователя лучше разделять на разные файлы.
- Проверка каждой функции сразу после её написания экономит время на поиск ошибок в дальнейшем.

## Урок 10.4. Тестирование и отладка готового проекта
**Время: 60 минут**

### Цель урока
Проверить работу итогового проекта на разных сценариях использования, найти и исправить ошибки.

### 1. Ручное тестирование по сценариям

Прежде чем считать проект готовым, важно пройти через все основные сценарии использования вручную:

- Добавление новой задачи — работает ли корректно?
- Отметка задачи как выполненной — обновляется ли статус?
- Удаление задачи — исчезает ли она из списка?
- Повторный запуск программы — сохраняются ли данные между запусками?

### 2. Проверка граничных случаев

Особое внимание стоит уделить нестандартным ситуациям:

- Что произойдёт, если попытаться удалить задачу с несуществующим id?
- Что произойдёт, если список задач пуст, а пользователь просит его показать?
- Что произойдёт, если пользователь введёт текст вместо числа при вводе id?
- Что произойдёт при первом запуске программы, когда файла с данными ещё не существует?

### 3. Написание простых unit-тестов (по желанию, для продвинутых)

Если позволяет время, полезно написать несколько автоматических тестов для ключевых функций логики, используя знания из Урока 9.3:

```python
import unittest
from tasks import add_task, complete_task, remove_task

class TestTaskTracker(unittest.TestCase):

    def test_add_task(self):
        tasks = []
        tasks = add_task(tasks, "Тестовая задача", "высокий")
        self.assertEqual(len(tasks), 1)
        self.assertEqual(tasks[0]["title"], "Тестовая задача")

    def test_complete_task(self):
        tasks = [{"id": 1, "title": "Задача", "priority": "средний", "done": False}]
        result = complete_task(tasks, 1)
        self.assertTrue(result)
        self.assertTrue(tasks[0]["done"])

    def test_remove_nonexistent_task(self):
        tasks = [{"id": 1, "title": "Задача", "priority": "средний", "done": False}]
        result = remove_task(tasks, 999)
        self.assertFalse(result)

if __name__ == "__main__":
    unittest.main()
```

### 4. Использование логирования для диагностики (по желанию)

При возникновении сложно воспроизводимых ошибок полезно временно добавить вызовы `logging` (Урок 9.2) в ключевые места кода, чтобы отследить, на каком именно шаге программа ведёт себя не так, как ожидалось.

### 5. Чек-лист готовности проекта

- [ ] Все функции из технического задания реализованы.
- [ ] Программа не «падает» при некорректном вводе пользователя.
- [ ] Данные корректно сохраняются и загружаются между запусками.
- [ ] Код разбит на логические файлы/функции, а не находится целиком в одном месте.
- [ ] В коде есть комментарии там, где логика неочевидна.
- [ ] Код в целом соответствует стилю PEP 8 (Урок 2.5).

### 6. Список найденных и исправленных ошибок

Рекомендуется вести короткий список: что было сломано → в чём была причина → как исправлено. Это не только помогает не забыть про уже найденные проблемы, но и станет полезным материалом для презентации проекта в следующем уроке.

### Резюме урока

- Тестирование проекта должно включать как типичные сценарии использования, так и граничные, нестандартные случаи.
- Чек-лист готовности помогает объективно оценить, насколько проект соответствует техническому заданию.
- Фиксация найденных и исправленных ошибок — полезная практика, пригодится при презентации проекта.

## Урок 10.5. Оформление и презентация проекта
**Время: 40 минут**

### Цель урока
Подготовить итоговый проект к демонстрации: оформить код и написать краткую презентацию проделанной работы.

### 1. Финальное оформление кода

Перед презентацией стоит ещё раз пройтись по коду и убедиться, что:

- Имена переменных и функций понятны и соответствуют PEP 8.
- Убраны отладочные `print()`, оставленные во время разработки (кроме осознанно оставленного логирования).
- Код не содержит закомментированных «мёртвых» кусков кода, оставшихся от экспериментов.
- В начале главного файла или в отдельном файле `README.md` есть краткое описание, что делает программа и как её запустить.

### 2. Пример простого README.md

```markdown
# Трекер задач

Консольное приложение для управления списком задач.

## Возможности
- Добавление, выполнение и удаление задач
- Сохранение данных между запусками в файл tasks.json

## Как запустить
1. Установите Python 3.10 или новее
2. Запустите: python main.py
```

### 3. Структура презентации проекта

Презентация итогового проекта (устная или в виде коротких слайдов) обычно строится по следующей схеме:

1. **Идея.** Какую проблему решает проект и почему она выбрана? (1 минута)
2. **Демонстрация.** Показать работу программы вживую — пройти основные сценарии использования. (2–3 минуты)
3. **Технические детали.** Какие темы курса использованы, какие были сложности при реализации, что получилось решить самостоятельно. (2 минуты)
4. **Возможные улучшения.** Что можно было бы добавить при наличии больше времени (например, графический интерфейс, работа с базой данных вместо файла и т.д.). (1 минута)

### 4. Частые вопросы, к которым стоит подготовиться

### 5. Значение итогового проекта для дальнейшего обучения

Итоговый проект — это не просто формальность для завершения курса, а первый опыт разработки полноценной, пусть и небольшой, программы «с нуля до готового результата». Такой опыт закладывает основу для дальнейшего изучения более сложных тем: веб-разработки, анализа данных или разработки более крупных приложений.

### 6. Что дальше после курса

После защиты итогового проекта можно двигаться в одном из следующих направлений (в зависимости от интересов):
- **Веб-разработка** — изучение Django или Flask.
- **Анализ данных** — изучение библиотек Pandas, NumPy, Matplotlib.
- **Автоматизация и скрипты** — более глубокое изучение работы с файлами, API, планировщиками задач.
- **Подготовка к собеседованиям** — решение алгоритмических задач, изучение структур данных на более глубоком уровне.

### Резюме урока

- Перед презентацией важно привести код в аккуратный вид и добавить краткое описание проекта.
- Презентация проекта строится по схеме: идея → демонстрация → технические детали → возможные улучшения.
- Итоговый проект — это первый полноценный опыт разработки готового приложения и хорошая база для дальнейшего изучения Python.
