## Урок 9.1. Отладка кода: ошибки и их поиск
**Время: 40 минут**

### Цель урока
Научиться читать сообщения об ошибках Python, находить причину сбоя программы и эффективно её исправлять.

### 1. Типы ошибок в Python

- **Синтаксические ошибки (SyntaxError)** — код написан с нарушением правил языка, программа вообще не запускается.
- **Ошибки времени выполнения (Runtime Errors)** — код синтаксически верен, но при выполнении возникает ошибка (например, деление на ноль).
- **Логические ошибки** — программа выполняется без сообщений об ошибке, но выдаёт неверный результат. Это самый сложный тип ошибок для поиска, так как Python никак не сигнализирует о них.

### 2. Как читать traceback

Когда в программе возникает необработанная ошибка, Python выводит **traceback** — трассировку стека вызовов, показывающую, где именно произошла ошибка:

```
Traceback (most recent call last):
  File "main.py", line 5, in <module>
    result = 10 / 0
ZeroDivisionError: division by zero
```

Читать traceback стоит снизу вверх: последняя строка (`ZeroDivisionError: division by zero`) говорит, какая именно ошибка произошла, а строки выше показывают, в каком файле и на какой строке это случилось.

### 3. Стратегии отладки

- **Чтение сообщения об ошибке.** Большинство ошибок Python описывает довольно точно — не стоит игнорировать текст ошибки.
- **Добавление print() в код.** Простой, но эффективный способ проверить промежуточные значения переменных на разных этапах выполнения программы.

```python
def calculate_average(numbers):
    print("Входные данные:", numbers)   # отладочный вывод
    total = sum(numbers)
    print("Сумма:", total)                 # отладочный вывод
    return total / len(numbers)
```

- **Использование отладчика (debugger).** В VS Code можно ставить точки останова (breakpoints) и пошагово выполнять код, наблюдая за значениями переменных в реальном времени — это удобнее, чем расставлять множество `print()`.

### 4. Частые причины ошибок у новичков

| Ошибка | Частая причина |
|---|---|
| `IndentationError` | неправильные или смешанные отступы |
| `NameError` | опечатка в имени переменной или использование до объявления |
| `TypeError` | попытка выполнить операцию над несовместимыми типами |
| `IndexError` | обращение к индексу списка, который не существует |
| `KeyError` | обращение к несуществующему ключу словаря |

### 5. Практический пример: поиск ошибки

```python
def average(numbers):
    total = 0
    for n in numbers:
        total += n
    return total / len(number)   # опечатка: number вместо numbers

print(average([1, 2, 3]))
```

Traceback покажет `NameError: name 'number' is not defined` — это сразу указывает на строку с опечаткой.

### 6. Работа с логическими ошибками

Логические ошибки не выдают сообщения, поэтому их ищут через сравнение ожидаемого и фактического результата:

```python
def is_even(n):
    return n % 2 == 1   # логическая ошибка: перепутано условие

print(is_even(4))   # False, хотя должно быть True
```

Такие ошибки выявляются через тестирование на конкретных примерах — именно этому посвящён следующий урок модуля.

### Резюме урока

- Traceback стоит читать снизу вверх — последняя строка указывает на тип и суть ошибки.
- `print()` для отладки и точки останова в IDE — два основных инструмента поиска ошибок.
- Логические ошибки не выдают сообщений и требуют внимательной проверки соответствия ожидаемого и реального результата.

## Урок 9.2. Логирование (модуль logging)
**Время: 30 минут**

### Цель урока
Познакомиться с модулем `logging` и понять, чем логирование лучше использования `print()` для отслеживания работы программы.

### 1. Зачем нужно логирование

По мере роста программы использование `print()` для отладки становится неудобным: сложно отличить обычный вывод от отладочного, невозможно легко отключить все отладочные сообщения сразу, нет информации о времени и важности сообщения. Модуль `logging` решает все эти проблемы.

### 2. Базовое использование logging

```python
import logging

logging.basicConfig(level=logging.INFO)

logging.debug("Отладочное сообщение")
logging.info("Информационное сообщение")
logging.warning("Предупреждение")
logging.error("Ошибка")
logging.critical("Критическая ошибка")
```

### 3. Уровни логирования

| Уровень | Когда используется |
|---|---|
| `DEBUG` | подробная информация для отладки |
| `INFO` | подтверждение нормальной работы программы |
| `WARNING` | что-то неожиданное, но программа продолжает работать |
| `ERROR` | серьёзная проблема, часть функционала не сработала |
| `CRITICAL` | критическая ошибка, программа может не продолжить работу |

По умолчанию отображаются сообщения уровня `WARNING` и выше — уровень можно изменить через параметр `level` в `basicConfig()`.

### 4. Форматирование сообщений лога

```python
import logging

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s - %(levelname)s - %(message)s"
)

logging.info("Программа запущена")
# 2026-07-12 10:00:00,123 - INFO - Программа запущена
```

### 5. Запись логов в файл

```python
import logging

logging.basicConfig(
    filename="app.log",
    level=logging.INFO,
    format="%(asctime)s - %(levelname)s - %(message)s"
)

logging.info("Это сообщение будет сохранено в файл app.log")
```

### 6. Практический пример: замена print() на logging

```python
import logging

logging.basicConfig(level=logging.INFO, format="%(asctime)s - %(levelname)s - %(message)s")

def read_file(filename):
    logging.info(f"Попытка открыть файл {filename}")
    try:
        with open(filename, "r") as file:
            content = file.read()
        logging.info("Файл успешно прочитан")
        return content
    except FileNotFoundError:
        logging.error(f"Файл {filename} не найден")
        return None

read_file("data.txt")
```

### Резюме урока

- Модуль `logging` — более гибкая и профессиональная альтернатива `print()` для отслеживания работы программы.
- Пять уровней важности сообщений: `DEBUG`, `INFO`, `WARNING`, `ERROR`, `CRITICAL`.
- Логи можно выводить в консоль или сохранять в отдельный файл для последующего анализа.

## Урок 9.3. Основы тестирования (модуль unittest)
**Время: 45 минут**

### Цель урока
Понять, зачем нужно автоматическое тестирование кода, и научиться писать простые тесты с помощью встроенного модуля unittest.

### 1. Зачем нужно тестирование

Автоматические тесты — это код, который проверяет корректность работы другого кода. Вместо того чтобы вручную запускать программу и проверять результат «на глаз» после каждого изменения, тесты позволяют быстро и надёжно убедиться, что всё работает как задумано — особенно важно при внесении изменений в уже написанный код.

### 2. Структура теста с unittest

`unittest` — модуль стандартной библиотеки Python для написания и запуска тестов.

```python
# calculator.py
def add(a, b):
    return a + b

def divide(a, b):
    if b == 0:
        raise ValueError("Деление на ноль недопустимо")
    return a / b
```

```python
# test_calculator.py
import unittest
from calculator import add, divide

class TestCalculator(unittest.TestCase):

    def test_add(self):
        self.assertEqual(add(2, 3), 5)
        self.assertEqual(add(-1, 1), 0)

    def test_divide(self):
        self.assertEqual(divide(10, 2), 5)

    def test_divide_by_zero(self):
        with self.assertRaises(ValueError):
            divide(10, 0)

if __name__ == "__main__":
    unittest.main()
```

### 3. Запуск тестов

```bash
python -m unittest test_calculator.py
```

Если все тесты прошли успешно, в терминале появится сообщение `OK`. Если какой-то тест не прошёл, unittest покажет, какой именно тест провалился и почему.

### 4. Основные методы проверки (assert)

| Метод | Проверяет |
|---|---|
| `assertEqual(a, b)` | что a равно b |
| `assertNotEqual(a, b)` | что a не равно b |
| `assertTrue(x)` | что x истинно |
| `assertFalse(x)` | что x ложно |
| `assertRaises(Ошибка)` | что код вызывает указанное исключение |
| `assertIsNone(x)` | что x равно None |

### 5. Хорошие практики написания тестов

- Каждый тест должен проверять одну конкретную вещь.
- Имена тестовых методов должны начинаться с `test_` и ясно описывать, что проверяется.
- Стоит проверять не только «нормальные» случаи, но и граничные (пустой список, ноль, отрицательные числа, некорректный тип данных).

### 6. Практический пример: тесты для функции is_prime

```python
# primes.py
def is_prime(n):
    if n < 2:
        return False
    for i in range(2, n):
        if n % i == 0:
            return False
    return True
```

```python
# test_primes.py
import unittest
from primes import is_prime

class TestIsPrime(unittest.TestCase):

    def test_prime_numbers(self):
        self.assertTrue(is_prime(2))
        self.assertTrue(is_prime(7))
        self.assertTrue(is_prime(13))

    def test_non_prime_numbers(self):
        self.assertFalse(is_prime(4))
        self.assertFalse(is_prime(9))

    def test_edge_cases(self):
        self.assertFalse(is_prime(0))
        self.assertFalse(is_prime(1))
        self.assertFalse(is_prime(-5))

if __name__ == "__main__":
    unittest.main()
```

### Резюме урока

- Автоматические тесты позволяют быстро проверять корректность кода при внесении изменений.
- Модуль `unittest` входит в стандартную библиотеку и позволяет писать тесты в виде классов с методами `test_...`.
- Важно проверять не только типичные случаи, но и граничные значения (нулевые, отрицательные, пустые данные).
